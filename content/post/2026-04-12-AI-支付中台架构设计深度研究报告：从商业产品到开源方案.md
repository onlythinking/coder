## 前言

支付系统是互联网金融的核心基础设施，也是业界公认的最复杂系统之一。它不仅涉及金钱的精确计算，还需要在高并发、高可用、强一致性之间找到微妙的平衡。

本文是一份深度研究报告，系统梳理了：

1. 主流商业支付平台的产品架构（Stripe、Adyen、蚂蚁、微信支付、京东支付）
2. 开源支付系统的技术方案与选型建议
3. 大厂支付中台的实战架构（阿里、字节、美团、京东、唯品会）
4. 支付中台核心模块的深度设计（幂等、分布式事务、对账、支付路由、风控）
5. 从0到1设计支付中台的整体思路与原则

预计阅读时间较长，建议收藏后分章节阅读。

## 第一部分：支付中台的基础认知

### 1.1 支付系统的层级划分

在讨论支付中台之前，需要先理解支付领域的层级划分。业界通常将支付系统分为五个层次：

| 层级 | 名称 | 职责 | 示例 |
|------|------|------|------|
| L1 | 业务前台 | 用户下单、收银台、支付方式选择 | 电商下单页、外卖结算页 |
| L2 | **支付中台** | 统一支付能力抽象、交易核心、账务中心 | 交易核心、支付路由、风控 |
| L3 | 支付网关 | 封装渠道协议（HTTP/API/ISO8583） | 支付宝直连适配器、银联直连接口 |
| L4 | 渠道层 | 银行、第三方支付机构 | 工行、招行、支付宝、微信支付、网联 |
| L5 | 人民银行的清算网络 | 资金最终清算 | 大额支付系统、小额支付系统、网联 |

**支付中台的定位**：位于业务前台与支付网关之间，负责通用的支付能力抽象，让业务前台无需关心底层渠道差异。

### 1.2 什么业务需要自建支付中台

不是所有公司都需要自建支付中台。决策树如下：

```
是否需要自建支付中台？
├── 单业务线、月交易额 < 1000万 → 直接接入第三方支付（Stripe/支付宝直连）
├── 多业务线、月交易额 1000万~1亿 → 接入聚合支付平台（Ping++、连连支付）
├── 多业务线、月交易额 > 1亿 → 考虑自建支付中台
└── 强监管场景（金融、跨境）→ 必须自建，需合规资质
```

**需要自建支付中台的典型特征**：
- 公司有多条业务线需要统一支付能力（如美团的外卖+酒旅+电商）
- 有资金分账/分润需求（平台型业务）
- 需要自主掌控支付数据（数据安全考量）
- 有跨境支付或自建钱包需求

### 1.3 支付中台与支付网关的区别

这是两个最容易混淆的概念：

**支付网关（Payment Gateway）**：专注于"如何与渠道通信"，本质是协议转换器。它处理签名、加密、报文封装、渠道路由等技术细节。常见的产品：支付宝网关SDK、Stripe API、银联在线支付接口。

**支付中台（Payment Middle Office）**：专注于"如何组织支付能力"，本质是能力聚合器。它在支付网关之上封装了路由、风控、账务、对账等通用能力。业务方只需要调用中台接口，不需要关心具体走哪个渠道。

简单说：**支付网关解决"怎么调渠道"的问题，支付中台解决"用什么渠道、怎么管钱"的问题。**

## 第二部分：全球主流支付平台产品调研

### 2.1 Stripe：API-First 的典范

Stripe 是全球最成功的支付基础设施公司之一，其估值曾一度超过900亿美元。它的核心价值主张是：**让开发者用最少的代码，接入全球最全的支付网络**。

#### 核心产品矩阵

```
Stripe 平台
├── Stripe Payments    → 基础支付（信用卡、借记卡、数字钱包）
├── Stripe Billing     → 订阅计费（月付、年付、分层定价）
├── Stripe Connect     → 平台分账（ marketplace 场景）
├── Stripe Atlas       → 公司注册（远程注册美国公司）
├── Stripe Terminal   → 线下POS
├── Stripe Radar       → 机器学习风控
└── Stripe Issuing     → 虚拟/实体卡发放
```

#### 技术架构亮点

**1. Idempotency Key 机制**

Stripe 所有写操作都支持 idempotency key，这是一套设计极为精巧的幂等机制：

```bash
# 请求示例
curl https://api.stripe.com/v1/payment_intents \
  -u sk_test_xxx: \
  -H "Idempotency-Key: iuaz7d8f2a9x8c7v6b5n4m3k2j1" \
  -d amount=2000 \
  -d currency=cny
```

特点：
- Key 由客户端生成（UUID格式）
- Key 有24小时有效期
- 返回结果包含 `original_key` 便于排查
- 重复请求返回相同结果，不重复扣款

**2. Webhook 异步通知体系**

Stripe 的 Webhook 设计是行业标杆：
- 事件驱动：支付状态变更、退款、拒付等都有独立事件
- 签名验证：每个 Webhook 都有 HMAC-SHA256 签名，防伪造
- 重试机制：若回调失败，Stripe 会自动按指数退避重试（最长72小时）
- 事件回放：可通过 API 重新触发历史事件

```json
// Webhook 事件示例
{
  "id": "evt_1234567890",
  "type": "payment_intent.succeeded",
  "data": {
    "object": {
      "id": "pi_xxx",
      "amount": 2000,
      "currency": "cny",
      "status": "succeeded"
    }
  }
}
```

**3. Dashboard 运营能力**

Stripe Dashboard 提供了近乎完美的运营体验：
- 实时交易监控
- 一键退款、对账
- 自动化结算报表
- 争议（Chargeback）管理

#### 开发者体验评价

| 维度 | 评分 | 说明 |
|------|------|------|
| API 设计 | ★★★★★ | RESTful，版本化，文档极详尽 |
| 多币种 | ★★★★★ | 支持135+货币，实时汇率转换 |
| 合规覆盖 | ★★★★★ | PCI DSS Level 1，PSD2，SCA |
| 落地难度 | ★★★☆☆ | 需海外公司开户，文档全英文 |

### 2.2 Adyen：企业级全球支付

Adyen 是欧洲老牌支付公司，2018年上市后成为支付领域市值最高的公司之一。与 Stripe 不同，Adyen 更侧重企业级大客户。

#### 核心客户

Netflix、Uber、Microsoft、Spotify、LVMH、欧莱雅——几乎所有我们熟知的全球互联网巨头都在用 Adyen。

#### 产品特点

**1. 全球直连银行**

Adyen 是少数支持直接接入各国本地清算网络的公司：
- 欧洲：SEPA Instant、Sofort、iDEAL、Bancontact
- 亚太：ALIPAY（直连）、WECHAT PAY（直连）
- 北美：ACH、RTP
- 拉美：Boleto、PEC

这种方式比走卡组织（Visa/Mastercard）费率更低，大客户谈判空间大。

**2. 线上线下一体化（Point-of-Sale）**

Adyen 的 POS 产品（Adyen for Platforms）打通了线上线下：
- 线上支付和线下扫码共享同一套商户账户
- 统一对账、统一结算
- 跨境电商 + 线下零售的融合场景天然适配

**3. 结算周期灵活**

支持 T+0（当日结算）到 T+30，结算周期可按商户配置，这在电商大促期间（如"账期延长"）非常有用。

#### 技术评价

| 维度 | 评分 | 说明 |
|------|------|------|
| API 设计 | ★★★★☆ | 功能全但文档较复杂 |
| 全球覆盖 | ★★★★★ | 直连银行覆盖广 |
| 企业特性 | ★★★★★ | 复杂分账、担保交易 |
| 定价 | ★★★☆☆ | 交易量大可谈折扣 |

### 2.3 蚂蚁集团（支付宝）：金融级架构

支付宝是全球最大的第三方支付平台，也是中国互联网金融的基础设施。蚂蚁集团的支付架构经过十五年双十一的锤炼，是业界分布式金融系统的天花板。

#### 产品矩阵（开放平台）

```
蚂蚁开放平台
├── 当面付         → 扫码支付（主扫、被扫）
├── APP支付        → 手机APP内调起支付宝
├── 手机网站支付   → WAP H5页面支付
├── 电脑网站支付   → PC网页支付
├── 小程序支付     → 支付宝小程序内支付
├── 资金授权       → 预授权冻结/解冻
├── 芝麻信用       → 免押金场景
├── 当面付的资金流向 → 服务商分润（直付通）
└── 支付后积分     → 会员积分能力
```

#### 直付通 vs 分账通

这是两个最容易混淆的产品：

**直付通**：资金直接由用户到商户，支付宝只做通道。商户自行承担资金结算风险。适用于：自营电商、自营服务平台。

**分账通**：资金先收到平台账户，再按比例分给各方。适用于： marketplace（淘宝、饿了么）、平台型业务。蚂蚁收取分账手续费（通常0.2%~0.6%）。

#### 技术架构亮点

**1. 分布式事务：TXC + Seata**

蚂蚁自研的分布式事务框架 TXC（Taobao Transaction Constructor），后开源为 Seata：

- **AT 模式**：自动补偿，全链路追踪，对业务无侵入
- **TCC 模式**：自定义补偿逻辑，性能更好
- **Saga 模式**：长流程编排，适合支付链路

```java
// Seata AT 模式示例
@GlobalTransactional
public void createOrder(OrderDTO order) {
    // 本地事务，Seata 自动记录 undo log
    orderService.insert(order);
    // 远程调用库存服务
    inventoryService.deduct(order.getProductId(), order.getCount());
}
```

**2. 账务分离架构**

蚂蚁的账务系统与交易系统完全分离：
- 交易系统记录流水
- 账务系统记录账户余额
- 两者通过事件异步同步
- 最终一致性通过对账保证

**3. 资金调度系统**

日终资金归集、夜间调拨、备付金管理——这是"断直连"后每家支付机构必须自建的模块。

### 2.4 微信支付：生态闭环的极致

微信支付凭借微信这个超级流量入口，在国内市场与支付宝二分天下（各占约45%市场份额）。

#### 核心产品

```
微信支付产品体系
├── Native支付     → 电脑网站二维码支付（生成链接，前台轮询）
├── JSAPI支付       → 微信内网页调起支付
├── 小程序支付      → 微信小程序内调起
├── APP支付         → 移动APP调起微信
├── H5支付          → 微信外部浏览器支付
├── 面部支付        → 刷脸设备
├── 付款码支付      → 展示付款码，被扫码
├── 企业付款到零钱  → 红包、佣金打款
├── 微信分账        → 平台分账（类似支付宝分账通）
└── 代金券/优惠券   → 营销工具
```

#### 技术特点

**1. 沙箱环境**

微信支付提供官方沙箱测试工具，商户可以：
- 获得一个测试商户号
- 模拟支付、退款、回调全流程
- 不需要真实资金

**2. 回调通知可靠性**

微信的支付结果通知设计：
- 同样的通知可能会发送多次，商户必须能够正确处理重复通知
- 通知机制：支付完成后，微信会把支付结果发送给商户，商户需要回复 success 或 fail
- 重试策略：若商户应答失败，微信会按一定频率重发通知，最多重发24小时

**这与 Stripe 的 Webhook 设计哲学一致——都是"至少一次"的语义。**

**3. 对账单下载**

微信提供每日对账单：
- 交易账单：逐笔交易明细
- 资金账单：账户资金变动
- 微信买单：门店收款汇总

### 2.5 京东支付：企业级 + 供应链金融

京东支付在集团内部的定位不仅是支付通道，更是供应链金融的入口。

#### 核心产品

- **京东支付（B2B）**：大额转账、企业钱包
- **聚合支付**：聚合微信/支付宝/银联
- **刷脸支付**：京东之家、7Fresh 门店
- **京东白条**：信用支付，供应链金融核心产品

#### 白条的账务设计

白条本质上是"虚拟信用卡"，其账务设计极为复杂：
- 用户授信额度管理（额度计算、动态调整）
- 分期手续费计算（费率和银行不同，更灵活）
- 坏账核销（催收系统对接）

## 第三部分：开源支付系统方案

### 3.1 自建 vs 使用开源 vs 外购

在讨论开源方案之前，先明确技术选型的决策框架：

```
支付系统技术选型决策树

评估维度：
1. 交易规模：QPS、峰值TPS
2. 业务复杂度：分账层数、到账时效要求
3. 合规要求：PCI DSS、PCI PA-DSS、资金安全
4. 开发团队能力：Java/Go 主力、是否有过金融系统经验
5. 时间成本：是否可以从0自建

选型建议：
├── 交易量小、团队小 → 直接用第三方SDK（Stripe/Ping++）
├── 有自研能力、交易量大 → 自建核心（参考开源架构）
└── 合规硬要求 → 必须外购或获得资质
```

### 3.2 开源支付系统一览

#### 3.2.1 Pay Java / Pay Spring

**定位**：Java 生态最活跃的开源支付轮子，封装了支付宝、微信、银联等国内主流渠道。

**特点**：
- 代码量适中，适合二次开发
- 社区活跃，文档较全
- 支持：支付宝（当面付/APP/手机网站/电脑网站/花呗/分期）、微信支付（Native/JSAPI/APP/H5/小程序）、银联（在线支付/企业网银/代付）

**适用场景**：中小型团队快速接入国内支付渠道，不建议直接用于生产核心系统。

#### 3.2.2 Ping++（商业 SaaS）

Ping++ 是国内头部的聚合支付 SaaS 平台，不是严格意义的开源，但值得在此讨论：

**产品架构**：
```
Ping++
├── 聚合支付 SDK（一次接入，接入所有渠道）
├── 统一商户后台
├── 资金结算（代付/代收）
├── 电子发票
└── 数据分析
```

**特点**：
- 一次对接，切换渠道零成本
- 适合：快速上线、业务初期不想自建
- 缺点：费用（0.6%~1% 交易手续费 + 结算费）

#### 3.2.3 Seata（分布式事务）

Seata 是阿里巴巴开源的分布式事务解决方案，源自内部 TXC 框架，是自建支付中台必备的基础设施。

**三种模式对比**：

| 模式 | 原理 | 性能影响 | 适用场景 |
|------|------|---------|---------|
| AT | 自动 undo log 补偿 | 中等 | 普通业务事务 |
| TCC | Try-Confirm-Cancel 手动编排 | 低 | 支付、资金冻结/解冻 |
| Saga | 长事务编排 | 低 | 跨系统长链路 |

**支付场景的推荐**：
- 账户余额操作用 **TCC**（性能好，控制精确）
- 订单 + 支付 + 发货等长链路用 **Saga**
- 简单查询和更新用 **AT**

#### 3.2.4 ShardingSphere（分库分表）

浙商银行等机构开源的 ShardingSphere 在大厂支付系统中广泛使用，用于解决单表数据量过大的问题。

**支付场景的分库分表策略**：

按交易日期分片（最常用）：
```
交易表 → 按日期分库/分表
2026-04-01 ~ 2026-04-07 → 表 t_trade_20260401 ~ t_trade_20260407
```

按商户 ID 分片（分账场景）：
```
商户账户表 → 按商户ID Hash分片
避免热点账户集中在一个分片
```

### 3.3 账务系统的开源方案

账务系统是支付中台最核心也最难的部分。业界几乎没有开箱即用的开源账务系统，以下是自建时的技术参考：

#### 复式记账的实现

会计恒等式：`资产 = 负债 + 所有者权益`

每笔交易需要记录借贷双方：

```sql
-- 账户表
CREATE TABLE account (
    id BIGINT PRIMARY KEY,
    account_no VARCHAR(32) UNIQUE,  -- 账户号
    account_type ENUM('CUSTOMER', 'MERCHANT', 'BANK', 'PLATFORM'),  -- 账户类型
    balance DECIMAL(20, 2) DEFAULT 0,  -- 余额
    frozen_balance DECIMAL(20, 2) DEFAULT 0,  -- 冻结金额
    version INT DEFAULT 0,  -- 乐观锁
    created_at DATETIME,
    updated_at DATETIME
);

-- 流水表（复式记账核心）
CREATE TABLE ledger_entry (
    id BIGINT PRIMARY KEY,
    tx_id VARCHAR(64) UNIQUE,  -- 关联交易ID
    account_no VARCHAR(32),  -- 账户
    entry_type ENUM('DEBIT', 'CREDIT'),  -- 借/贷
    amount DECIMAL(20, 2),  -- 金额
    balance_before DECIMAL(20, 2),  -- 变动前余额
    balance_after DECIMAL(20, 2),  -- 变动后余额
    remark VARCHAR(256),  -- 摘要
    created_at DATETIME
);

-- 交易表
CREATE TABLE transaction (
    id BIGINT PRIMARY KEY,
    tx_no VARCHAR(64) UNIQUE,  -- 交易单号
    tx_type ENUM('PAY', 'REFUND', 'TRANSFER', 'WITHDRAW'),  -- 交易类型
    amount DECIMAL(20, 2),
    status ENUM('PENDING', 'SUCCESS', 'FAILED'),
    channel VARCHAR(32),  -- 渠道标识
    merchant_id VARCHAR(64),
    created_at DATETIME,
    completed_at DATETIME
);
```

**复式记账的核心理念**：每笔交易必须同时记录"钱从哪里来"和"钱到哪里去"，借贷必相等。这不仅满足会计要求，也为对账提供了完整的数据基础。

## 第四部分：大厂支付中台架构实践

### 4.1 阿里巴巴：十五年双十一的锤炼

阿里巴巴（支付宝）的支付架构是中国支付行业的天花板。蚂蚁集团的 SOFA 框架、OceanBase 数据库、分布式事务能力，都源自双十一的极限压力测试。

#### 核心架构原则

**1. 单元化架构（Cell-based Architecture）**

单元化是蚂蚁应对海量并发的核心方案：

```
整个支付系统按商户/用户分为 N 个单元
每个单元包含完整的：交易 + 账务 + 渠道 能力
单元之间不直接调用，通过同步机制保证最终一致

好处：
- 故障隔离：单个单元故障不影响其他单元
- 水平扩展：容量不够时增加单元即可
- 容灾：异地多活，每个城市多个单元
```

**2. 三地五中心容灾**

蚂蚁的容灾方案是"三地五中心"：
- 三个城市：上海、杭州、张北
- 每城市至少两个独立数据中心
- 任意一个城市整体故障，0 RPO 切换

**3. 账务系统设计**

账务系统是蚂蚁最引以为傲的系统之一：

```
账务核心设计原则：
1. 账务与交易完全解耦
   - 交易系统：记录交易流水（交易单维度）
   - 账务系统：记录账户变化（账户维度）
   - 两者通过 MQ 异步同步

2. 日切机制
   - 每日 00:00 进行日切
   - 生成日终快照
   - 日切期间停止账户变动操作

3. 余额对账
   - 每笔交易后，账户余额 = 历史余额 + 本笔发生额
   - 实时更新，防止并发超扣
```

### 4.2 字节跳动：支付平台的单元化实践

字节跳动的支付中台起步较晚，但架构设计上吸收了蚂蚁等前辈的经验，并在此基础上做了创新。

#### 核心架构

**1. 支付路由的多级设计**

```
一级路由：按业务线（抖音/TikTok/电商）
  ↓
二级路由：按支付方式（余额/银行卡/第三方）
  ↓
三级路由：按渠道（招行/工行/支付宝/微信）
  ↓
四级路由：按渠道质量（成功率/费率）动态选择
```

这种多级路由的好处：
- 新业务接入只需适配一级路由规则
- 渠道故障时，只需在对应层级摘除，不影响其他渠道
- 可以做 AB 测试验证路由效果

**2. 交易核心的 CQRS 模式**

字节支付核心采用了 CQRS（Command Query Responsibility Segregation）：

```
写路径（Command）：
用户支付请求 → 交易核心写模型 → 幂等校验 → 状态机 → MQ

读路径（Query）：
交易状态查询 → 读模型（Elasticsearch）→ 实时返回

好处：
- 写入和读取资源隔离，互不影响
- 读模型可以针对性优化（ES 查询性能极佳）
- 写入模型专注事务一致性
```

### 4.3 美团支付：多业务线抽象

美团是中国互联网公司中业务线最多的公司之一（外卖、酒旅、到店、电商、金融），其支付中台的核心挑战是：**如何用一套系统服务这么多差异化业务？**

#### 核心设计：支付内核 + 业务插件

```
美团支付平台
├── 支付内核（通用能力）
│   ├── 交易状态机
│   ├── 幂等控制
│   ├── 支付路由
│   ├── 风控规则引擎
│   └── 统一账务接口
│
└── 业务插件层（差异化扩展）
    ├── 外卖插件：红包 + 积分 + 抵用券
    ├── 酒旅插件：担保交易（入住前冻结，离店后结算）
    ├── 电商插件：分阶段付款（定金 + 尾款）
    └── 到店插件：多次核销（一次购买，多次消费）
```

**这种设计的精髓**：核心系统保持稳定，差异化能力通过插件机制扩展。核心系统不感知业务差异，所有差异都在插件层处理。

#### 组合支付的抽象

组合支付（多种支付方式叠加）是美团等平台型业务的常见需求：

```json
{
  "组合支付请求": {
    "total_amount": 100.00,
    "payment_methods": [
      {"type": "COUPON", "amount": 20.00},
      {"type": "BALANCE", "amount": 30.00},
      {"type": "WECHAT_PAY", "amount": 50.00}
    ]
  }
}
```

拆单逻辑：
- 优先使用自有资产（优惠券、余额）
- 剩余金额选择主支付渠道
- 每种支付方式独立幂等，独立回调
- 全部成功则交易成功；任一失败则整体回滚

### 4.4 京东支付：账务与交易分离

京东的支付架构核心哲学是"账务优先"——在任何异常情况下，以账务系统的记录为准。

#### 架构特点

```
京东支付核心
├── 交易系统（只负责状态流转）
│   └── 订单创建 → 支付中 → 交易完成/关闭
│
├── 账务系统（只负责资金变动）
│   ├── 账户管理（余额、冻结）
│   ├── 资金流水（每笔变化都有记录）
│   └── 试算平衡（借贷必相等）
│
└── 对账系统（最终仲裁）
    └── 以账务系统为准，交易系统服从对账结果
```

**这与阿里的"账务交易分离"理念一致，但京东更极端——交易系统几乎不存储任何资金相关的汇总数据。**

### 4.5 唯品会：电商大促支付保障

唯品会作为特卖电商，大促期间的支付峰值是日常的50倍以上。其支付中台的经验主要是：

#### 大促保障措施

**1. 支付链路异步化**

```
同步路径（已废弃）：
用户点击支付 → 前台等待 → 渠道返回 → 显示结果
（用户等待时间长，体验差，且超时难以处理）

异步路径（推荐）：
用户点击支付 → 前台立即返回"支付中" → 渠道异步回调 → 主动查询补齐 → 状态推送
```

**2. 渠道降级预案**

```
正常情况：支付宝（80%）+ 微信支付（20%）

预案A（支付宝繁忙）：
支付宝（30%）+ 微信支付（40%）+ 银行卡（30%）

预案B（支付宝不可用）：
微信支付（60%）+ 银行卡（40%）

预案C（所有第三方不可用）：
银行卡直连（需提前和银行申请应急通道）
```

**3. 对账前置化**

大促期间，对账从 T+1（日终）提前到 T+0（准实时）：
- 每小时对账一次
- 发现差异立即告警
- 大促结束后再做一次完整日终对账

### 4.6 PayPal：国际化参考

PayPal 作为全球覆盖最广的支付平台之一，其架构有以下几点值得参考：

**1. 账户模型：两步验证 + 动态风控**

PayPal 的账户体系设计：
- 每笔支付都有基于设备和行为的风控评分
- 可疑交易需要额外验证（密码、短信、人脸）
- 账户限额分级（未认证 / 已认证 / 企业认证）

**2. 跨境结算的货币转换**

PayPal 支持 25 种货币，但中间转换损失较大：
- 实时汇率（每天更新）
- 货币转换费（2.5%~3%）
- 这也是 Stripe 等竞争对手的主要差异化点

**3. 争议处理机制（Dispute Resolution）**

```
买家发起争议（180天内）
  ↓
PayPal 介入调解（冻结资金）
  ↓
卖家申诉（提供发货凭证、签收证明）
  ↓
```

这套机制是 Stripe Radar、京东售后等产品的参考原型。

## 第五部分：支付中台核心模块深度设计

### 5.1 幂等性设计

幂等是支付系统的基石。没有幂等，就没有安全的重试。

#### 幂等的四个层次

**层次一：单次请求的幂等（最基础）**

```java
// 伪代码：支付请求幂等
public PayResult pay(PayRequest request) {
    // 1. 检查是否已经执行过
    Payment existed = paymentService.findByIdempotentKey(request.getIdempotentKey());
    if (existed != null) {
        return PayResult.success(existed); // 返回已有结果，不重复执行
    }

    // 2. 执行支付
    Payment payment = doPay(request);

    // 3. 记录幂等Key
    paymentService.saveWithIdempotentKey(payment, request.getIdempotentKey());

    return PayResult.success(payment);
}
```

**层次二：与渠道交互的幂等（更复杂）**

渠道方（支付宝、微信）返回超时——此时无法判断是"支付成功但回调丢失"还是"支付失败"。处理方式：

```java
public PayResult handleChannelCallback(ChannelCallbackRequest request) {
    // 1. 优先查本地交易记录
    Payment localPayment = paymentService.findByChannelTxId(request.getChannelTxId());

    if (localPayment != null && localPayment.isSuccess()) {
        return PayResult.success(localPayment); // 本地已处理，直接返回
    }

    // 2. 本地无记录，查询渠道（需要渠道提供查询接口）
    ChannelQueryResult result = channelService.query(request.getChannelTxId());

    if (result.isPaySuccess()) {
        // 3. 确认成功，本地落库（幂等）
        paymentService.confirmSuccess(request.getChannelTxId(), result.getAmount());
        return PayResult.success();
    }

    return PayResult.fail("支付失败");
}
```

**层次三：退款操作的幂等（最容易出错）**

退款是最需要幂等的场景，因为用户会因网络原因多次点击：

```sql
-- refund_request 表（退款请求）
CREATE TABLE refund_request (
    id BIGINT PRIMARY KEY,
    refund_no VARCHAR(64) UNIQUE,  -- 退款单号，客户端生成
    original_trade_no VARCHAR(64),  -- 原交易单号
    refund_amount DECIMAL(20, 2),
    status ENUM('PENDING', 'SUCCESS', 'FAILED'),
    channel_refund_no VARCHAR(64),  -- 渠道退款单号
    created_at DATETIME,
    completed_at DATETIME
);
```

```java
public RefundResult refund(RefundRequest request) {
    // 1. 幂等检查：用退款单号查重
    RefundRequest existed = refundService.findByRefundNo(request.getRefundNo());
    if (existed != null) {
        return existed.isSuccess() ? RefundResult.success() : RefundResult.fail(existed.getFailReason());
    }

    // 2. 检查原交易是否可退（未退完、退款中、已退款）
    Payment original = paymentService.findByTradeNo(request.getOriginalTradeNo());
    if (!original.canRefund(request.getAmount())) {
        return RefundResult.fail("超出可退款金额");
    }

    // 3. 向渠道发起退款（如果渠道报错，要有明确的错误码区分）
    try {
        ChannelRefundResult result = channelService.refund(request);
        // 4. 更新本地状态
        refundService.updateSuccess(request.getRefundNo(), result.getChannelRefundNo());
        return RefundResult.success();
    } catch (ChannelRefundFailedException e) {
        if (e.isRetryable()) {
            // 可重试的错误（如网络超时），记录后可重试
            refundService.updateFailed(request.getRefundNo(), e.getMessage());
            throw e;
        } else {
            // 不可重试的错误（如金额超限），直接标记失败
            refundService.updateFailed(request.getRefundNo(), e.getMessage());
            return RefundResult.fail(e.getMessage());
        }
    }
}
```

**层次四：分账操作的幂等**

分账（Split Payment）是平台型业务的核心需求，通常一对多：

```java
public SplitResult split(SplitRequest request) {
    // 分账天然是幂等的，因为每笔分账明细都有唯一ID
    for (SplitDetail detail : request.getDetails()) {
        SplitDetail existed = splitService.findByDetailId(detail.getDetailId());
        if (existed != null) {
            continue; // 已分账，跳过
        }
        doSplit(detail);
    }
    return SplitResult.success();
}
```

#### 幂等存储的选型

| 存储 | 适用场景 | 特点 |
|------|---------|------|
| MySQL（唯一索引）| 强一致性要求，所有请求必须落库 | 性能一般，可靠性高 |
| Redis + MySQL 双写 | 高并发写入，退款等核心操作 | Redis 拦截，MySQL 持久化 |
| Redis 唯一Key | 短时幂等（如5分钟内重复提交拦截）| 速度快，不支持长周期追溯 |

**最佳实践**：核心支付链路用 MySQL 唯一索引幂等；查询类轮询用 Redis。

### 5.2 分布式事务设计

支付链路涉及多个系统：交易系统 → 账务系统 → 渠道网关 → MQ → 对账系统。任何一步失败都需要回滚。

#### 三种主流方案对比

| 方案 | 原理 | 性能 | 业务侵入 | 适用场景 |
|------|------|------|---------|---------|
| 2PC/3PC | 协调者全局锁定 | 低 | 高 | 不推荐用于支付 |
| TCC | 业务补偿（Try/Confirm/Cancel）| 高 | 中 | 支付、资金冻结 |
| Saga | 编排式长事务 | 高 | 低 | 跨系统长链路 |

#### TCC 在支付中的最佳实践

TCC 是支付场景最推荐的分布式事务方案。以"支付 + 分账"为例：

```java
// Try 阶段：预留资源
@TwoPhaseBusiness
public void payAndSplit(PayAndSplitRequest request) {
    // T - 冻结用户账户（不减余额，冻结金额 +1）
    accountService.freeze(request.getUserId(), request.getAmount());

    // T - 向渠道发起支付
    channelPayService.tryPay(request);

    // T - 生成分账记录（状态为待确认）
    splitService.createPendingSplits(request);
}

// Confirm 阶段：确认执行
public void confirm(BusinessActionContext context) {
    // C - 扣减用户余额，冻结金额 -1
    accountService.confirmDeduct(context.getUserId(), context.getAmount());

    // C - 更新分账记录为已分账
    splitService.confirmSplits(context.getSplitIds());

    // C - 通知渠道（如需）
    channelPayService.confirm(context);
}

// Cancel 阶段：回滚
public void cancel(BusinessActionContext context) {
    // C - 解冻用户账户
    accountService.unfreeze(context.getUserId(), context.getAmount());

    // C - 更新分账记录为已取消
    splitService.cancelSplits(context.getSplitIds());

    // C - 通知渠道撤销（如渠道支持）
    channelPayService.cancel(context);
}
```

**TCC 的关键原则**：
- Try 操作必须快速完成，不能有外部 IO（禁止在 Try 里调渠道）
- Confirm/Cancel 必须是幂等的
- 补偿逻辑要尽可能覆盖所有异常路径

#### RocketMQ 事务消息

RocketMQ 的事务消息是"本地消息表"模式的进阶版：

```java
// 发送方
@Transactional
public void sendPaymentMQ(Payment payment) {
    // 1. 本地事务：写入本地库
    paymentDAO.insert(payment);

    // 2. 发送半消息（RocketMQ 事务消息）
    //  HalfMessage 状态为 PENDING，不会被消费
    rocketMQTemplate.sendMessageInTransaction(
        "payment_topic",
        MessageBuilder.withPayload(payment).build(),
        context
    );
}

// RocketMQ 回调：本地事务状态查询
@Override
public LocalTransactionState checkLocalTransaction(MessageExt msg) {
    Payment payment = paymentDAO.findById(msg.getKeys());
    if (payment == null) {
        return LocalTransactionState.ROLLBACK_MESSAGE;
    }
    return payment.isSuccess() ? LocalTransactionState.COMMIT_MESSAGE : LocalTransactionState.ROLLBACK_MESSAGE;
}
```

### 5.3 对账系统设计

对账是支付系统的"最后一道防线"。它验证整个链路的数据一致性，是发现 Bug、防止资金损失的最后机会。

#### 对账的分层设计

```
对账分层
├── 渠道对账：与支付宝/微信/银行的交易流水核对
│   目的：确保我方记录与渠道记录一致
│
├── 内部对账：交易系统 vs 账务系统
│   目的：确保交易状态和资金变动一致
│
├── 资金对账：账户余额 vs 交易流水汇总
│   目的：确保账户余额 = 历史 + 流水
│
└── 汇总对账：日终汇总 vs 实时监控
    目的：发现异常波动
```

#### 渠道对账的标准流程

```python
# 伪代码：支付宝渠道对账
def reconcile_alipay(date):
    # 1. 下载对账文件
    alipay_file = alipay.download_reconciliation(date)

    # 2. 解析文件
    alipay_records = parse(alipay_file)  # [{trade_no, amount, status, time}]

    # 3. 查询本地交易
    local_records = db.query(
        "SELECT trade_no, amount, status FROM trade WHERE date = %s AND channel = 'ALIPAY'",
        date
    )

    # 4. 比对
    alipay_map = {r['trade_no']: r for r in alipay_records}
    local_map = {r['trade_no']: r for r in local_records}

    all_trade_nos = set(alipay_map.keys()) | set(local_map.keys())

    diff_report = []
    for trade_no in all_trade_nos:
        alipay_rec = alipay_map.get(trade_no)
        local_rec = local_map.get(trade_no)

        if alipay_rec and not local_rec:
            # 长款：渠道有，本地没有（渠道已付，本地未记账）
            diff_report.append(('LONG', trade_no, alipay_rec['amount'], '渠道已付，本地未记账'))
        elif local_rec and not alipay_rec:
            # 短款：本地有，渠道没有（本地已记账，渠道未付）
            diff_report.append(('SHORT', trade_no, local_rec['amount'], '本地已记账，渠道未付'))
        elif alipay_rec['amount'] != local_rec['amount']:
            # 金额差异
            diff_report.append(('AMOUNT_DIFF', trade_no,

                f"本地{local_rec['status']} vs 渠道{alipay_rec['status']}"))

    # 5. 生成差错单

        error_ticket_service.create(diff)
```

### 5.5 风控系统设计

支付风控的核心目标：**在用户无感知的情况下，拦住尽可能多的欺诈交易。**

#### 风控的三道防线

```
第一道：实时风控（< 100ms）
  规则：单笔限额、频次限制、黑白名单、设备指纹、IP画像

第二道：准实时风控（分钟级）
  规则：异常行为检测（同IP多账号、非常用设备）

第三道：离线风控（T+N）
  规则：洗钱检测、套现识别、套利识别
```

### 6.1 设计原则总结

基于前面的调研和案例，这里总结支付中台设计的十大原则：

| 序号 | 原则 | 说明 |
|------|------|------|
| 1 | 账务与交易分离 | 账务是事实，交易是过程；以账务为准 |
| 2 | 幂等优先 | 所有外部调用必须是幂等的 |
| 3 | 异步化是常态 | 支付链路全异步，用户无感知等待 |
| 4 | 对账是最后防线 | 任何异常最终通过对账兜底 |
| 5 | 渠道与业务解耦 | 新增渠道只需适配器，不改核心 |
| 6 | 路由可观测 | 所有路由决策可追溯、可回滚 |
| 7 | 风控无感知 | 正常用户不应感知风控存在 |
| 8 | 容灾有预案 | 每个渠道都有降级链路 |
| 9 | 合规不妥协 | 资金安全 > 用户体验 > 系统性能 |
| 10 | 数据不丢失 | 任何一笔交易都不能被遗忘 |
