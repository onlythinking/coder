# Stack Overflow 开发者调查：66% 程序员被 AI 编程工具坑惨了

你花三分钟让 AI 写了一个用户认证模块，跑起来报错了。转头一看，AI 写了一个完全不存在的 Python 标准库函数。你花二十分钟找到错误，改了三行代码，感觉"还好"——但如果当初自己写，可能只需要十分钟。

这个场景，正在成为全球开发者的日常。

## 一个让 Copilot 用户沉默的数字

Stack Overflow 刚刚发布了年度开发者调查报告，样本量超过五万人，覆盖全球主要编程社区。核心结论只有一个，足以让每一个天天用 AI 写代码的人重新审视自己的工作流：

**66% 的受访开发者表示，AI 编程工具生成的代码带来的返工率，高于自己手写的代码。**

不是 66% 的人觉得 AI 不好用。是 66% 的人被 AI 坑过，发现自己修 AI 的 bug 比从头写还慢。

这不是在说 AI 工具没有价值。而是在说，当 AI 帮你"提高效率"的时候，它同时在制造另一种低效——你需要花更多时间去理解它写了什么、为什么这样写、以及它哪里写错了。这个隐性成本，在每一篇 AI 编程工具的测评里，从来没有人告诉你。

## 代码产量暴涨，bug 产量也在暴涨

报告给出了几个具体维度，每个维度都值得我们停下来想一想。

**简单场景里，AI 工具确实香。** 写一个 CRUD 接口、生成一段正则匹配、处理一下 JSON 序列化——这类重复性模板代码，Copilot、Codeium 们的表现足够好。开发者普遍反馈："这种活儿让 AI 干，我喝茶等结果就行。"

但一旦进入复杂业务逻辑，AI 的幻觉问题开始显著增加。

所谓幻觉，就是 AI 会一本正经地编造不存在的东西。它会拼贴出一个看起来像那么回事的 API 签名，但这个函数根本不存在；它会在 Python 里调用一个 JavaScript 的库函数；它会在你完全没意识到的情况下，用一个过时的语法写了一整段逻辑。

更普遍的场景是这样的：AI 生成了一段代码，跑起来报错了。开发者花了半小时理解 AI 的思路——为什么它会这样写？它是从哪个上下文里学的这个模式？——然后再花半小时修正。最后发现，这个"半小时理解 + 半小时修正"的工作量，比从头手写这段代码还要慢。

这不是个案。报告里有一句来自受访者的原话，被引用了无数次：

> "我把 AI 生成的代码放进生产环境之前，必须先在脑子里把它『翻译』成我自己的逻辑。这个翻译过程比我手写还慢。"

这句话说出了 AI 编程工具最被忽视的一个问题：**你不是在写代码，你是在审核 AI 的代码。而审核，往往比创作更需要深度。**

## 布罗克曼的 80% 与 Stack Overflow 的 66%

就在这份报告发布的同时，OpenAI 总裁格雷格·布罗克曼（Greg Brockman）披露了另一组数据：OpenAI 内部工程师使用 AI 工具编写的代码占总代码量的比例，已从两成猛增至八成。

这个数字和 Stack Overflow 的调查放在一起看，像是一个精心设计的对比实验：

OpenAI 的工程师们——全球最懂 AI 模型运作原理的那批人——有 80% 的代码是 AI 写的；而全球五万普通开发者里，66% 被 AI 代码坑过。

区别在哪里？OpenAI 工程师深度理解 AI 模型的能力边界，知道什么场景可以让它发挥，什么场景要收回控制权。普通开发者在享受 AI 加速的同时，往往在用"信任"代替了"审核"。

这就是差距的来源。

## GitHub 出了什么事

调查报告还呼应了另一个值得关注的现象：AI 编程热潮推动 GitHub 在近几个月频繁出现平台级故障，GitHub 被迫启动了底层基础设施重构。

一个工具平台出问题，通常是因为用的人太多了。但这次不太一样。

故障日志和社区反馈指向了一个更具体的原因：AI 工具让"提交代码"变得太容易了。以前一个工程师一天提交 5-10 次 PR；现在有了 Copilot Cursor，同样的工程师一天可以轻松产出 30-50 个 commit。代码质量参差不齐，PR 量翻了三到五倍，issue 量、CI 构建量随之暴涨。GitHub 的底层存储没有问题，是上层的流程系统在替 AI 化浪潮买单。

这不是 GitHub 一家的问题。这是整个开发生态在 AI 化转型中正在经历的调整期的缩影。

## 66% 的教训：工具进化，能力也要进化

Stack Overflow 的报告揭示了一个根本性的错位：

**AI 编程工具解决的是「写代码的速度」问题，但没有解决「写对代码的能力」问题。**

速度和能力，不是一回事。

对于初学者或中级开发者，AI 生成代码时，他们往往缺乏足够的判断力来识别错误——因为能够识别错误的前提，是你知道正确的是什么。AI 给了你一碗看起来很好的饭，但你不知道里面有没有你不该吃的东西。

对于高级开发者，AI 在他们熟悉的领域可能帮不上太多忙——因为这个领域你本来就能写，AI 的产出和你脑子里想的差不多，没有增量价值。反而在陌生领域，AI 生成的内容里会混入大量你没有能力审核的噪音。

那么，到底该怎么用 AI 编程工具？

一个务实的原则：**把 AI 当作「速度放大器」，而不是「能力替代品」。**

具体来说，AI 工具适合的场景是：重复性模板代码、语法糖补全、陌生的 API 签名查询、文档快速生成、单测基础骨架、以及你本来就要删掉的探索性代码。

不适合交给 AI 的场景是：核心业务逻辑、涉及资金或安全的代码、复杂状态机设计、需要深度领域知识的模块、以及你未来还要维护超过三个月的任何代码。

66% 的返工率，是一个足够清晰的信号：在某些场景下，AI 工具的效率是负的。识别这个边界，比无条件拥抱更重要。

## 最后

2025 年是「氛围编程」（vibe coding）元年。柯林斯词典把这两个词选为年度词汇，Karpathy 本人——这个词的命名人——说了一句话：AI Coding 没门槛，可部署环节却真难。

门槛降低了，但山还在那里。

工具进化得越来越快，用工具的人也需要跟着进化。不然，你以为你在提效，其实是在用 AI 的速度挖更深的坑。

**下次让 AI 帮你写代码的时候，先问自己一个问题：这段代码，如果出事了，我能不能自己修？**

如果答案是否定的，那这段代码，就不该让 AI 写。
