## 一个真实的深夜盗刷

凌晨三点，用户老王收到一条短信："您的账户在深圳消费9,800元，购买冰箱一台。"老王人在北京，从没买过冰箱。

查交易记录：凌晨三点零五分，一笔9,800元的支付请求通过微信支付完成。支付设备是一台从未见过的安卓手机，IP地址在深圳。支付完成后，钱已经转到了某个商户账户。

更严重的是：老王的账户在这笔交易之前，已经被攻击者绑定了一张新银行卡——这张卡不是老王的，但支付平台的风控系统没有发现异常。

这就是一次典型的账户盗刷。攻击者拿到老王的账户信息后，在凌晨三点这个风控盲区，用新设备+新IP+新银行卡的组合，完成了一笔大额交易。

风控系统要做的，就是在这笔交易完成之前，把它拦截下来。

## 一、什么是支付风控

支付风控：在用户发起支付到资金最终清算的整个链路中，识别并拦截欺诈交易、套现行为、薅羊毛等风险操作的技术系统。

风控和业务是矛盾的：风控越严，用户体验越差（验证码多、步骤多、失败率高）；风控越松，损失越高。风控系统的目标是在**准确率（抓对的/抓错的好人）**和**召回率（抓到的/所有坏人）**之间找到平衡点。

两个核心指标：

| 指标 | 含义 | 公式 |
|------|------|------|
| **准确率** | 拦截的异常中，真的有问题的比例 | TP / (TP + FP) |
| **召回率** | 所有异常中，被成功拦截的比例 | TP / (TP + FN) |

- 准确率低 → 误伤多，好用户被拦截，客诉飙升
- 召回率低 → 漏抓多，欺诈分子进来，资金损失

**不同阶段的侧重点**：
- 业务早期：召回率优先，宁可误伤也要守住风险底线
- 业务稳定期：准确率优先，减少误伤提升用户体验
- 大促期间：召回率优先，守住资损红线

## 二、风控三道防线

风控不是单点拦截，而是分层防御：

### 第一道：实时风控（<100ms响应）

在支付请求进入的毫秒级时间内，完成风险判断。

核心特征：快、准、轻。不做深度计算，只判断明显的异常信号。

```java
public RiskDecision evaluateRealtime(PaymentRequest request) {
    // 1. 名单黑名单匹配（基于哈希表查找，O(1)复杂度）
    if (blackListService.isBlackUser(request.getUserId())) {
        return RiskDecision.REJECT;  // 直接拒绝
    }
    if (blackListService.isBlackDevice(request.getDeviceId())) {
        return RiskDecision.REJECT;
    }
    if (blackListService.isBlackIP(request.getIp())) {
        return RiskDecision.REJECT;
    }

    // 2. 规则引擎打分（每条规则贡献风险分，累计总分）
    RuleResult ruleResult = ruleEngine.evaluate(request);
    int ruleScore = ruleResult.getRiskScore();  // 0~100+

    // 3. 机器学习模型评分（0~1，越高越可疑，映射到0~100）
    double modelScore = mlModel.predict(request);
    int mlScore = (int) (modelScore * 100);  // 0.9 → 90分

    // 4. 综合分数 = 规则分×0.6 + 模型分×0.4（规则优先，毕竟可解释）
    int totalScore = (int) (ruleScore * 0.6 + mlScore * 0.4);

    // 5. 分数→风险等级→决策（与5.1节风险等级表格对应）
    if (totalScore > 90) {
        return RiskDecision.REJECT;  // 极高风险：直接拦截
    } else if (totalScore > 70) {
        return RiskDecision.REVIEW;  // 高风险：人工外呼
    } else if (totalScore > 30) {
        return RiskDecision.VERIFY;  // 中风险：增强验证（短信/人脸）
    }
    return RiskDecision.PASS;       // 低风险：直接放行
}
```

### 第二道：准实时风控（<5秒响应）

支付完成后几分钟内，分析交易行为模式。如果实时风控漏了，这层来补。

典型场景：用户支付时正常，但支付完成后行为异常（比如5分钟内又来一笔大额交易，或者资金快速转出）。

```python
# 准实时风控：基于时间窗口的行为分析
def evaluate_quasi_realtime(transaction):
    user_id = transaction.user_id

    # 窗口1：5分钟内交易笔数
    recent_count = redis.get(f"txn_count:{user_id}:5min")
    if recent_count and int(recent_count) > 5:
        return RiskDecision.SUSPECT("5分钟内交易超过5笔")

    # 窗口2：1小时内累计金额
    recent_amount = redis.get(f"txn_amount:{user_id}:1h")
    total = (int(recent_amount or 0)) + transaction.amount
    if total > user_service.get_user_daily_limit(user_id):
        return RiskDecision.SUSPECT("1小时累计金额超限")

    # 窗口3：资金流向分析（钱进了哪个商户，是否是可疑商户）
    if merchant_service.is_high_risk(transaction.merchant_id):
        return RiskDecision.SUSPECT("资金流向高风险商户")

    # 更新计数器
    redis.incr(f"txn_count:{user_id}:5min")
    redis.expire(f"txn_count:{user_id}:5min", 300)
    redis.incrby(f"txn_amount:{user_id}:1h", int(transaction.amount))
    redis.expire(f"txn_amount:{user_id}:1h", 3600)

    return RiskDecision.PASS
```

### 第三道：离线风控（次日分析）

每天跑批分析全量交易，挖掘新的欺诈模式，更新规则和模型。

- 挖掘新欺诈模式：聚类分析、异常检测
- 标签回溯：用历史数据训练模型，提高召回率
- 名单更新：把新发现的欺诈用户加入黑名单

```python
# 离线风控：欺诈模式挖掘
def offline_risk_analysis(date):
    yesterday_txns = transaction_service.get_by_date(date)

    # 1. 聚类分析：找出异常行为群
    # （同一设备/同一IP/同一商户的批量交易）
    clusters = clustering.detect_anomaly_clusters(yesterday_txns)

    for cluster in clusters:
        if cluster.size > 10 and cluster.amount_std < 100:
            # 金额高度一致 → 疑似机器刷单
            flag_cluster_as_fraud(cluster)
            update_blacklist_from_cluster(cluster)

    # 2. 异常检测：单个用户行为异常
    for user in yesterday_txns.unique_users():
        features = extract_user_features(user, date)
        if isolation_forest.is_anomaly(features, threshold=0.8):
            mark_user_for_review(user)

    # 3. 模型更新：用昨日标签回溯训练
    labels = get_labels_from_manual_review(date - 7)  # 7天前的工单结果
    model.update(labels)
```

## 三、风控规则引擎设计

规则引擎是风控系统的核心——把业务经验转化成可配置、可追溯的规则。

### 3.1 规则结构

一条规则由三部分组成：**条件**（什么情况下触发）、**阈值**（触发参数）、**动作**（触发后做什么）。

```java
public class RiskRule {
    private String code;           // 规则编号
    private String description;   // 规则描述
    private Condition condition;   // 触发条件
    private RiskAction action;     // 触发动作：PASS / REVIEW / REJECT
    private int riskScore;         // 风险分增量（供模型汇总用）
    private boolean enabled;       // 是否启用
}

// 条件类型枚举
public enum ConditionType {
    USER_DEVICE_NEW,       // 用户使用新设备
    USER_IP_CHANGED,       // 用户IP变化
    AMOUNT_ABOVE_THRESHOLD,// 金额超阈值
    TXN_FREQUENCY_HIGH,   // 交易频率过高
    TIME_OF_DAY_ABNORMAL, // 交易时间异常（如凌晨）
    RECIPIENT_HIGH_RISK,   // 收款方高风险
    USER_ACCOUNT_AGE_LOW,  // 账户注册时间短
}
```

### 3.2 规则示例

```java
// 规则1：新设备+新IP+新银行卡 → 高风险
RiskRule rule_new_device_combination = RiskRule.builder()
    .code("R_001")
    .description("新设备+新IP+新银行卡同时出现")
    .condition(c -> c.isNewDevice() && c.isNewIP() && c.isNewBankCard())
    .action(RiskAction.REJECT)
    .riskScore(60)
    .enabled(true)
    .build();

// 规则2：账户注册7天内首次大额交易（超5000元）→ 需审核
RiskRule rule_new_user_large_txn = RiskRule.builder()
    .code("R_002")
    .description("新用户注册7天内首次交易超5000元")
    .condition(c -> c.getUserAgeDays() < 7
                 && c.getAmount().compareTo(new BigDecimal("5000")) > 0)
    .action(RiskAction.REVIEW)
    .riskScore(30)
    .enabled(true)
    .build();

// 规则3：凌晨2点-5点交易且金额超2000元 → 标记
RiskRule rule_late_night = RiskRule.builder()
    .code("R_003")
    .description("凌晨2-5点交易超2000元")
    .condition(c -> c.getHour() >= 2 && c.getHour() <= 5
                 && c.getAmount().compareTo(new BigDecimal("2000")) > 0)
    .action(RiskAction.REVIEW)
    .riskScore(20)
    .enabled(true)
    .build();
```

### 3.3 规则热更新

规则不能靠发版来更新，要支持运营人员实时调整。

```java
public class RuleEngine {
    private LoadingCache<String, List<RiskRule>> rulesCache = Caffeine.newBuilder()
        .refreshAfterWrite(1, TimeUnit.MINUTES)  // 每分钟刷新规则
        .build(key -> ruleDao.getAllEnabledRules());

    public RuleResult evaluate(PaymentRequest request) {
        List<RiskRule> rules = rulesCache.get("current");

        int totalRiskScore = 0;
        RiskAction finalAction = RiskAction.PASS;

        for (RiskRule rule : rules) {
            if (rule.getCondition().matches(request)) {
                totalRiskScore += rule.getRiskScore();

                // 规则动作优先级：REJECT > REVIEW > PASS
                if (rule.getAction() == RiskAction.REJECT) {
                    return RuleResult.builder()
                        .action(RiskAction.REJECT)
                        .riskScore(totalRiskScore)
                        .triggeredRule(rule.getCode())
                        .build();
                }
                if (rule.getAction() == RiskAction.REVIEW) {
                    finalAction = RiskAction.REVIEW;  // 不覆盖REJECT
                }
            }
        }

        return RuleResult.builder()
            .action(finalAction)
            .riskScore(totalRiskScore)
            .triggeredRule(null)
            .build();
    }

    // 运营在后台改了规则，缓存失效，下一次读取自动加载新规则
    public void reloadRules() {
        rulesCache.invalidate("current");
    }
}
```

## 四、机器学习模型

规则引擎能覆盖已知的欺诈模式，但无法发现"未知的未知"。机器学习模型来解决这个问题。

### 4.1 特征工程

风控模型的特征分四类：

| 特征类型 | 示例 | 数据来源 |
|----------|------|---------|
| 用户特征 | 账户年龄、历史交易次数、平均交易金额 | 用户中心 |
| 设备特征 | 设备指纹、IP归属地、是否为模拟器 | 设备指纹SDK |
| 行为特征 | 操作间隔、滑动速度、输入节奏 | 前端埋点 |
| 关系特征 | 和其他欺诈用户的关联度、共用设备数 | 图数据库 |

```python
def extract_features(request: PaymentRequest, user: User) -> dict:
    return {
        # 用户特征
        'user_age_days': (datetime.now() - user.registered_at).days,
        'user_total_txn_count': user.transaction_count,
        'user_avg_txn_amount': user.average_transaction_amount,
        'user_has_real_name': user.is_verified,

        # 设备特征
        'device_is_simulator': request.device.is_simulator,
        'device_app_version': request.device.app_version,
        'ip_is_proxy': request.ip.is_proxy(),  # IP是否代理

        # 行为特征
        'operation_interval': request.operation_time - request.page_load_time,
        'device_fingerprint_match': request.device.fingerprint_match_rate,

        # 关系特征
        'user_device_relation_count': graph_db.count_device_users(request.device.id),
        'user_ip_relation_count': graph_db.count_ip_users(request.ip),
    }
```

### 4.2 模型选型

| 模型 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **XGBoost/LightGBM** | 二分类欺诈检测 | 可解释性强，特征重要性可分析 | 需要大量标注数据 |
| **Isolation Forest** | 异常检测（无监督） | 不需要标签，能发现新型欺诈 | 准确率不稳定 |
| **Graph Neural Network** | 团伙欺诈检测 | 能发现关联关系 | 训练成本高，复杂 |

### 4.3 模型解释性（对抗性要求）

风控模型必须能解释"为什么这个用户被拒绝"——因为需要给运营和用户提供理由，也因为监管要求（の風控决策必须可解释）。

```python
# SHAP值解释：每个特征对最终决策的贡献
import shap

explainer = shap.TreeExplainer(xgb_model)
shap_values = explainer.shap_values(features)

# 输出Top3风险因素，供运营参考
top_risk_factors = pd.DataFrame({
    'feature': feature_names,
    'shap_value': shap_values[0],
    'feature_value': features.values
}).sort_values('shap_value', ascending=False).head(3)

print(f"该笔交易Top3风险因素：")
for _, row in top_risk_factors.iterrows():
    print(f"  {row['feature']}={row['feature_value']:.2f}, 贡献风险分={row['shap_value']:.2f}")
```

## 五、风控与业务体验的平衡

风控做严了，用户投诉；做松了，资金损失。平衡点是动态的。

### 5.1 风险等级与处理方式

| 风险等级 | 分值区间 | 处理方式 | 用户感知 |
|----------|---------|---------|---------|
| **低** | 0-30分 | 直接放行 | 无感知 |
| **中** | 30-70分 | 增强验证（短信/人脸） | 支付需多一步验证 |
| **高** | 70-90分 | 人工外呼确认 | 支付延迟，客服联系 |
| **极高** | >90分 | 直接拦截，账号风控 | 账号受限，需申诉 |

### 5.2 灰度发布规则变更

规则调整（尤其是收紧规则）必须灰度，不能一刀切。

```java
public RiskDecision evaluate(PaymentRequest request, String ruleVersion) {
    // 灰度策略：10%流量用新规则，90%用旧规则
    // 注意：不用Python3的hash()（随机化），用MD5取模实现稳定的流量分桶
    if ("v2_new".equals(ruleVersion) && md5Bucket(request.getUserId(), 10) == 0) {
        return ruleEngineV2.evaluate(request);
    }
    return ruleEngineV1.evaluate(request);
}

// MD5分桶：同一用户每次路由到同一桶，跨进程/重启稳定
private int md5Bucket(String userId, int bucketCount) {
    try {
        MessageDigest md = MessageDigest.getInstance("MD5");
        byte[] digest = md.digest(userId.getBytes("UTF-8"));
        int hash = ((digest[0] & 0xFF) << 8) | (digest[1] & 0xFF);
        return Math.abs(hash % bucketCount);
    } catch (NoSuchAlgorithmException e) {
        return 0;
    }
}
```

灰度期间同时监控两个指标：新规则拦截率 vs 旧规则拦截率，误伤率变化。如果新规则召回率提升但准确率下降过多，暂缓全量上线。

## 六、三个真实的踩坑案例

### 案例1：新用户优惠被薅走

平台推出"新用户首单减10元"活动。上线一周，发现有10万个账户用了这个优惠——但这些账户大多是"真手机号、假用户"（虚拟号/物联网卡），用完优惠就注销。

**教训**：新用户风控不能只靠"账户注册时间"判断。要叠加：手机号段黑名单（虚拟号/物联网卡）、设备黑名单（同一设备注册多个账户）、IP黑名单（同一IP大量新用户注册）。优惠券薅羊毛的根因是注册环节的风控漏洞，不是支付环节的问题。

---

### 案例2：正常用户被误伤，客诉飙升

某次风控规则升级，把"账户注册30天内、交易金额超过1000元且IP归属地与常用地址不同"的判定阈值从"REVIEW"改成了"REJECT"。结果影响了一大批正常用户（出差、旅游、换城市）的支付，客诉量当天翻了3倍。

**教训**：收紧规则一定要灰度。哪怕规则逻辑看起来没问题，也要先10%流量试跑，确认误伤率可控再全量。

---

### 案例3：机器学习模型被对抗样本绕过

风控团队上线了一个XGBoost模型，识别率很高。但两周后欺诈分子发现了：他们在交易时故意在特征里注入"正常行为"（比如操作间隔故意变长、模拟正常人的输入节奏），模型准确率从92%掉到了61%。

**教训**：机器学习模型需要持续迭代。监控模型准确率下降趋势，当准确率跌破阈值（比如85%）时触发告警。同时定期用最新的欺诈样本重新训练模型（至少每月一次），不能一劳永逸。

## 七、结语

支付风控的本质是在"用户体验"和"资金安全"之间动态平衡。

三个核心原则：

1. **分层防御**：实时→准实时→离线，单一环节的疏漏不会导致全面失守
2. **规则+模型双轨**：规则覆盖已知风险，模型发现未知风险，互相补充
3. **可解释性**：每一次风控决策都必须能追溯到具体的规则或特征，这是监管要求，也是运营优化的基础

最后记住：风控系统的误判成本由用户承担，漏抓成本由平台承担。两者不是对称的——风控宁可误伤，不能漏抓。
