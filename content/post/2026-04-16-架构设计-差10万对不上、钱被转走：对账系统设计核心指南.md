## 一个真实的账不平事故

某天早上九点，财务发现：微信支付后台显示昨晚23:00有一笔10万元的聚合支付订单，但系统里这笔订单的状态是"失败"。

查日志：用户确实支付了10万，微信也回调了，系统也记录了回调——但因为回调里一个字段格式和预期不符，解析时报错，订单被标记为失败。这笔钱的账就这样悬着，直到财务对账时才发现。

这就是典型的"单边账"：渠道有数据，商户没数据，或者反过来。

账不平的损失有两层：一层是资金损失（10万）；另一层是账目混乱——如果没被发现，月底对账时这笔钱会变成"悬账"，要么追不回，要么要花大量人力去查。

对账系统要解决的，就是这个问题的早期发现和自动化处理。

---

## 一、什么是对账

狭义的对账：商户系统记录的交易流水，和渠道（微信/支付宝/银行）提供的交易记录，逐条比对，找出差异。

广义的对账：还包括：
- 资金对账：账户余额变动是否和交易记录一致
- 账务对账：内部账户（用户余额、积分、保证金）的变动是否正确
- 清算对账：和渠道结算周期结束后，确认最终清算金额

日常说的"对账"通常是第一种——交易级对账，也是最复杂的一种。

**三个维度**：

| 维度 | 描述 | 典型问题 |
|------|------|---------|
| 交易对账 | 系统流水 vs 渠道流水逐笔比对 | 单边账、数据格式差异、重复扣款 |
| 资金对账 | 账户余额 vs 交易流水汇总 | 平台垫资、资金被挪用 |
| 清算对账 | 结算周期 vs 实际到账金额 | 手续费争议、渠道拒付 |

## 二、对账的两种模式：实时 vs 批次

### 实时对账

用户在支付完成后，系统立即和渠道确认交易状态。适用于高风险交易（金额超过阈值、新用户首单、大额充值）。

优点：问题发现早，损失可以在用户还在线时追回或拦截。
缺点：渠道API有延迟（通常1-5秒），实时对账会增加用户等待时间；接口调用成本高。

```java
public void afterPaymentSuccess(Payment payment) {
    // 异步实时对账，不阻塞用户
    CompletableFuture.runAsync(() -> {
        try {
            // 等3秒，让渠道回调先到
            Thread.sleep(3000);

            // 查询渠道真实状态
            ChannelQueryResponse query = channelService.queryPayment(
                payment.getChannelOrderNo()
            );

            // 比对：系统状态 vs 渠道状态
            if (!payment.getStatus().equals(query.getStatus())) {
                // 状态不一致，告警+人工介入
                alertService.send("账务异常：系统状态={}，渠道状态={}",
                    payment.getStatus(), query.getStatus());
                reconciliationService.createException(payment, query);
            }
        } catch (Exception e) {
            log.error("实时对账失败", e);
        }
    });
}
```

**适用场景**：大额交易（单笔超过5000元）、新用户首单、涉及风控拦截的可疑交易

### 批次对账

每天固定时间（比如凌晨2点），拉取昨天的完整交易文件，和系统流水批量比对。适用于绝大多数交易——量大、金额中等、不需要实时确认。

这是对账系统的核心战场。

```python
# 批次对账核心流程（Python伪代码）
def reconcile(date):
    # 1. 从渠道下载对账文件
    channel_file = channel_file_service.download(
        date, channel='wechat')

    # 2. 从系统导出交易流水
    system_txns = transaction_service.export(date)

    # 3. 构建系统交易Map（以渠道订单号为Key）
    system_map = {tx.channel_order_no: tx for tx in system_txns}

    # 4. 逐行解析渠道文件，比对
    exceptions = []
    for record in channel_file.parse():
        sys_txn = system_map.get(record.order_no)

        if sys_txn is None:
            # 渠道有，系统没有 → 疑似盗刷或掉单
            exceptions.append(ReconException(
                type='CHANNEL_ONLY',
                channel_order_no=record.order_no,
                amount=record.amount))
        elif sys_txn.amount != record.amount:
            # 金额不符
            exceptions.append(ReconException(
                type='AMOUNT_MISMATCH',
                channel_order_no=record.order_no,
                sys_amount=sys_txn.amount,
                channel_amount=record.amount))
        else:
            # 匹配，标记为已对账
            mark_reconciled(sys_txn.id)

    # 5. 检查系统有、渠道没有的记录
    for txn in system_txns:
        if not is_reconciled(txn.id):
            exceptions.append(ReconException(
                type='SYSTEM_ONLY',
                order_no=txn.order_no,
                amount=txn.amount))

    # 6. 汇总上报警告
    alert_service.send_reconciliation_report(date, exceptions)
```

**适用场景**：日常交易对账、渠道日结算文件核对、所有非实时要求的账务核对

## 三、批次对账的核心流程

批次对账不是简单的两边文件逐行比对，背后有完整的状态机和处理流程。

### 3.1 文件获取：最容易出问题的第一步

微信支付、支付宝、银联的日结算文件，获取方式各不相同，而且都不稳定。

- **微信支付**：通过SFTP推送文件，但推送时间不固定（通常凌晨2:00-6:00），且可能推送失败后重试
- **支付宝**：通过API主动拉取，有频率限制，大商户文件可能分片
- **银联**：通过专线或文件服务器拉取，文件命名格式不统一

```java
public Optional<ChannelFile> fetchChannelFile(ChannelEnum channel, LocalDate date) {
    // 1. 先查本地缓存，文件获取有重试机制
    Optional<ChannelFile> cached = localCache.get(channel, date);
    if (cached.isPresent()) {
        return cached;
    }

    // 2. 尝试从渠道获取
    try {
        ChannelFile file = channel.fetchReconFile(date);

        // 3. 校验文件完整性（行数、金额汇总、文件大小）
        validateFileIntegrity(file);

        // 4. 存入本地缓存
        localCache.put(channel, date, file);
        return Optional.of(file);

    } catch (FileNotReadyException e) {
        // 渠道文件还没准备好，记录并等待下一次重试
        log.warn("渠道{}文件未就绪，date={}，等待重试", channel, date);
        return Optional.empty();
    } catch (FileFetchException e) {
        // 获取失败，告警
        alertService.send("渠道{}对账文件获取失败: {}", channel, e.getMessage());
        return Optional.empty();
    }
}
```

**常见问题**：
1. 渠道文件推送延迟，导致对账任务启动时文件还不存在
2. 文件获取后内容为空（渠道Bug，文件还没生成完就推送了）
3. 文件格式和文档不符（渠道悄悄改了格式，没有通知）

**解决方案**：文件获取要重试3次，每次间隔递增（5分钟、15分钟、30分钟）。超过重试次数则告警，人工介入。

---

### 3.2 数据清洗：格式不标准是常态

渠道文件的格式和你的系统几乎肯定不一样。字段命名、金额单位、时间格式、字符编码都能出幺蛾子。

```java
public ChannelTransaction parseWechatRow(String row) {
    // 微信对账文件示例行：
    // 2024-01-15 23:00:00, 20240115230000, Wx123456, Order001, SUCC, 1000, CNY
    // 字段：交易时间、订单号、渠道订单号、商户订单号、状态、金额(分)、币种

    String[] fields = row.split(",");
    return ChannelTransaction.builder()
        .transactionTime(parseTime(fields[0]))
        .channelOrderNo(fields[2].trim())
        .merchantOrderNo(fields[3].trim())
        .status(mapWechatStatus(fields[4]))  // 渠道状态→系统状态映射
        .amount(YuanToFen(fields[5]))       // 渠道用"分"，系统用"元
        .currency(fields[6].trim())
        .build();
}
```

**三个高频坑**：

1. **金额单位**：微信对账文件金额单位是"分"，系统数据库通常用"元"或"分"——必须统一，否则金额永远对不上
2. **状态映射**：渠道状态码和系统状态码不是一一对应的（"SUCCESS"vs"TRADE_SUCCESS"，"FAIL"vs"PAYER_CLOSED"）
3. **时间格式**：渠道文件和系统日志的时区可能不一致（UTC vs 北京时间），导致同一条记录在两边的时间戳差8小时

---

### 3.3 比对策略：宽对 vs 严对

比对策略决定了你能发现多少问题，也决定了误报率。

**宽对**：以商户订单号为基准，只要金额和状态匹配，就算对上。适用于渠道回调及时、数据质量高的场景。

**严对**：不仅要金额和状态一致，还要交易时间在合理时间范围内（比如渠道规定的清算时效内，不同渠道规定不同），金额精确到分。适用于风控要求高的场景。

```java
public ReconciliationResult match(ChannelTransaction ch, SystemTransaction sys) {
    // 基础匹配：金额必须相等
    if (!ch.getAmount().equals(sys.getAmount())) {
        return ReconciliationResult.MISMATCH_AMOUNT;
    }

    // 严对模式：检查时间差
    if (strictMode) {
        long timeDiff = Math.abs(
            ch.getTransactionTime().toEpochMilli()
            - sys.getTransactionTime().toEpochMilli()
        );
        if (timeDiff > TimeUnit.MINUTES.toMillis(5)) {
            return ReconciliationResult.MISMATCH_TIME;
        }
    }

    // 状态匹配（渠道SUCCESS → 系统SUCCESS/REFUNDED）
    if (!isStatusConsistent(ch.getStatus(), sys.getStatus())) {
        return ReconciliationResult.MISMATCH_STATUS;
    }

    return ReconciliationResult.MATCHED;
}
```

## 四、差错处理四象限

对账的核心价值不在于"发现差异"，而在于"处理差异"。对不上的账分成四类，每类处理方式不同。

| 差错类型 | 描述 | 系统有/渠道有 | 处理策略 |
|----------|------|--------------|---------|
| **短款** | 渠道有，系统没有 | 渠道有+系统无 | 盗刷/掉单？先查系统日志，确认则补录订单+告警 |
| **长款** | 系统有，渠道没有 | 系统有+渠道无 | 虚假交易？先查支付流水，确认未扣款则冲正 |
| **金额差** | 两边都有，金额不符 | 金额不一致 | 直接告警，人工介入，禁止自动处理 |
| **状态差** | 两边都有，状态不符 | 状态不一致 | 区分：系统失败但渠道成功（重灾区）vs 系统成功但渠道失败 |

### 4.1 短款（渠道有，系统没有）

这是最危险的一种——用户的钱被扣了，但你的系统不知道。

**排查步骤**：
1. 先在系统日志里搜这个渠道订单号，看是否有回调记录
2. 如果有回调但入库失败（数据库报错），则修复数据，补录订单
3. 如果没有回调记录，可能是用户支付后网络断了，回调没发出来——用渠道订单号查单接口确认状态
4. 确认为成功交易后，补录订单，人工确认货物是否已发放（如果已发货，则钱最终会到账；如果没发货，要联系用户或冻结货物）

**自动处理条件**：系统日志有回调记录、金额低于阈值（5000元，基于客诉率和风控成本的经验值）、非新用户。反之，一律人工处理。

---

### 4.2 长款（系统有，渠道没有）

钱没扣，但系统以为扣了。这种情况比短款好处理，但也可能是虚假订单。

**排查步骤**：
1. 查支付渠道的原始日志，看这笔交易是否真的被发起过
2. 如果是支付渠道报错导致没扣钱，确认后冲正订单（状态改为失败）
3. 如果是用户伪造支付（很少见但存在），走风控+安全流程

**特别说明**：长款中有一部分是"掉单"——用户支付后关掉了页面，没等到回调就离开了。这种情况建议先用渠道查单接口确认状态，不要直接标记为失败，否则用户可能联系客服说"我明明付了钱"。

---

### 4.3 金额差

金额差是最敏感的差错类型。少给钱还好解释，多给钱可能涉及合规问题。

**原则**：金额差一律不许自动处理，必须人工审核。金额超过1000元的金额差，当天必须处理。

---

### 4.4 状态差

这是最常见的一种差错。系统状态是"失败"，但渠道状态是"成功"——或者反过来。

"系统失败、渠道成功"是最常见的场景，根因通常是：
1. 渠道回调了，但系统处理时数据库报错（库存不足、账户异常等）
2. 系统处理成功，但回写状态时网络抖动，状态没更新成功
3. 超时了，渠道已经处理，但系统以为失败了

```java
public void handleStatusMismatch(SystemTransaction sys, ChannelTransaction ch) {
    // 场景1：系统失败，渠道成功 → 需要补发货
    if ("FAIL".equals(sys.getStatus()) && "SUCCESS".equals(ch.getStatus())) {
        if (sys.getAmount().compareTo(new BigDecimal("5000")) < 0) {
            // 金额低，可疑度低 → 自动补发货
            orderService.recoverOrder(sys.getOrderNo());
            alertService.send("自动恢复订单：orderNo={}", sys.getOrderNo());
        } else {
            // 金额高 → 人工审核
            reconciliationService.createException(sys, ch);
        }
    }

    // 场景2：系统成功，渠道失败 → 两种可能：(1)用户被重复扣款 (2)系统误报成功
    if ("SUCCESS".equals(sys.getStatus()) && "FAIL".equals(ch.getStatus())) {
        // 第一步：用渠道订单号查单确认真实状态
        ChannelQueryResponse realQuery = channelService.queryPayment(sys.getChannelOrderNo());

        if ("SUCCESS".equals(realQuery.getStatus())) {
            // 真的被扣了两次款 → 退第二笔
            reconciliationService.refundSecondCharge(sys, realQuery);
            alertService.send("重复扣款已退款：orderNo={}", sys.getOrderNo());
        } else {
            // 系统误报成功，实际没扣钱 → 冲正订单
            orderService.reverseOrder(sys.getOrderNo());
            alertService.send("订单已冲正（系统误报）：orderNo={}", sys.getOrderNo());
        }
    }
}
```

## 五、技术实现的关键设计

### 5.1 对账文件存储

对账文件是法律凭证，至少要保留5年（支付行业合规要求）。

```java
// 文件存储策略：本地+异地备份
public void storeReconFile(ChannelEnum channel, LocalDate date, String content) {
    // 1. 存本地（方便查询）
    String localPath = String.format(
        "/data/reconciliation/%s/%s/%s.txt",
        channel.name(),
        date.getYear(),
        date.toString()
    );
    fileStorage.write(localPath, content);

    // 2. 存OSS（异地备份，满足合规要求）
    ossClient.putObject(
        "reconciliation-backup",
        "recon/{}/{}/{}.txt".formatted(channel, date.getYear(), date),
        new ByteArrayInputStream(content.getBytes())
    );

    // 3. 设置OSS生命周期，5年后自动删除（满足人行要求）
    ossClient.setObjectLifecycle("reconciliation-backup", "recon/", 365 * 5);
}
```

**重要**：不要只存数据库，对账文件要存原始文本（不是解析后的数据），因为解析逻辑可能出错，原始文件是唯一的真相来源。

---

### 5.2 对账状态机

每笔交易的对账状态是一个独立的状态机：

```
PENDING → CHECKING → MATCHED
                  → EXCEPTION_SHORTAGE（短款）
                  → EXCEPTION_OVERAGE（长款）
                  → EXCEPTION_AMOUNT（金额差）
                  → EXCEPTION_STATUS（状态差）
```

状态流转规则：
- PENDING → CHECKING：开始对账
- CHECKING → MATCHED：两边完全一致
- EXCEPTION_* → MATCHED：人工修复后状态变更为已对上
- EXCEPTION_* → WRITTEN_OFF：确认无法追回，坏账核销

---

### 5.3 对账任务的调度

对账任务必须可靠执行，推荐使用分布式调度（XXL-Job/ElasticJob），避免单机调度因机器重启而漏跑。

```java
@XxlJob("dailyReconciliationJob")
public ReturnT<String> execute() {
    LocalDate yesterday = LocalDate.now().minusDays(1);

    log.info("开始执行日对账，日期={}", yesterday);

    // 1. 获取所有需要对账的渠道
    List<ChannelEnum> channels = channelService.getActiveChannels();

    // 2. 并行对账（不同渠道互不影响）
    List<CompletableFuture<ReconciliationReport>> futures = channels.stream()
        .map(ch -> CompletableFuture.supplyAsync(() -> reconcile(ch, yesterday)))
        .collect(Collectors.toList());

    // 3. 汇总结果
    ReconciliationReport total = futures.stream()
        .map(CompletableFuture::join)
        .reduce(ReconciliationReport.ZERO, ReconciliationReport::merge);

    // 4. 发送报告
    alertService.sendReconciliationSummary(total);

    // 5. 标记对账完成
    reconciliationService.markCompleted(yesterday);

    return ReturnT.SUCCESS;
}
```

## 六、三个真实的踩坑案例

### 案例1：微信对账文件"幽灵行"

某次对账发现，有200多笔订单系统查不到对应的支付记录，但微信对账文件里有这些订单，金额也不大（每笔几十元）。查了很久，发现是微信在特定活动期间，用了另一套清算通道，这些订单的渠道订单号前缀和正常订单不一样，系统解析时漏掉了。

**教训**：对账文件格式不是铁板一块，渠道会在不通知的情况下切换清算批次。对账系统要能处理"未知的未知"——即文件里有完全无法解析的行，不要直接报错退出，而是记录到"未识别记录"里，定期分析。

---

### 案例2：凌晨2点对账，文件还没来

对账任务凌晨2点启动，但微信的文件要凌晨4点才推送完成。导致每天有2小时的账是"半成品"状态。

**教训**：对账任务不能写死时间，要依赖文件存在性检测。文件没到就不跑对账，而不是到了时间就跑对账。用"文件拉取重试队列"替代"定时触发"，每次重试间隔递增。

---

### 案例3：已核销的短款又出现了

人工处理了一笔短款，补录了订单，标记为"已处理"。但下个月对账时，同一笔订单又出现了——原来渠道对账文件是按"清算周期"生成的，同一笔交易可能出现在多个清算文件里（因为清算周期之间有重叠）。

**教训**：对账差错处理要有幂等性——同一笔差错只处理一次，用"差错记录ID"做唯一键，而不是"订单号+日期"。并且要区分"已核对"和"已修复"，已核对但未修复的差错不能重复告警。

## 七、结语

对账系统做得好不好，不在于你的代码多复杂，而在于：

1. **文件获取稳不稳**——重试机制、超时处理、告警到位
2. **数据清洗够不够糙**——字段格式、单位、编码全部统一，有解析不了的行要能跳过不丢数
3. **差错分类清不清**——四类差错处理策略不同，混在一起处理必然出错
4. **合规留存有没有**——原始文件5年留存，这是监管要求，不是可选项

对账是支付系统里少有的"没有技术难点，但处处是业务坑"的地方。技术只是工具，核心是对业务的理解和对异常情况的敬畏。
