## 一个真实的事故

凌晨两点，客服涌入大量投诉：用户反映被扣了两次钱。

查日志，发现了这样的轨迹：

- 用户在收银台点击"立即支付"
- 微信唤起后，用户手机卡了一下
- 用户以为没反应，又点了一次
- 同一笔订单，微信支付回调了两次
- 系统没有做幂等处理，两次都去查库存、插订单、发货

最终：100多单重复扣款，涉及金额超过8万元。

这不是技术不行，是幂等性设计缺失。

---

## 一、幂等性到底是什么

大多数人对幂等性的理解是：**"防止用户重复操作"**。

这个理解对，但不够精确。

数学上的幂等是指：`f(x) = f(f(f(x)))`，无论执行多少次，结果相同。

在支付系统里，幂等性的准确定义是：

> **同一个业务请求，无论因为网络重试、用户误操作、还是系统故障，被执行了多少次，最终的业务结果必须与执行一次保持一致。**

关键词是"业务结果"，而不是"请求次数"。

两个请求同时来，都是扣100元库存。系统必须保证：无论处理过程多曲折，用户账户只扣一次，库存只减一次。

**这里有一个反直觉的陷阱**：幂等性不是"防重复提交"，而是"保证最终一致性"。

防重复提交是前端行为，靠按钮禁用、Token防抖实现。但前端防不住微信支付回调重发，防不住消息队列延迟堆积，防不住运维重跑历史任务。

## 二、三种主流实现方案

### 方案一：Token + 幂等表

**原理**：前端在创建订单时先向服务端申请一个唯一Token（类似全局唯一ID），随支付请求一起携带。服务端将Token作为唯一键写入幂等表，利用数据库唯一键约束保证不重复处理。

```
前端                    服务端                      数据库
  |                        |                          |
  |--申请Token----------->|                          |
  |<--Token=uuid-xxx------|                          |
  |                        |                          |
  |--支付请求(Token)------>|  INSERT INTO idempotent   |
  |                        |  (token, status) VALUES  |
  |                        |  (uuid-xxx, 'PROCESSING')|
  |                        |  --> 成功，执行业务       |
  |                        |                          |
  |                        |  |--支付回调(Token)------>|
  |                        |  | INSERT ... uuid-xxx   |
  |                        |  | --> 唯一键冲突！      |
  |                        |  | --> 查询原状态，返回成功
```

**适用场景**：高并发下单创建、用户主动发起的支付请求
**代价**：每次支付多一次数据库写入；Token过期需要重新申请

**代码示例**：

```java
public PayResult doPay(PayRequest request) {
    // 1. 幂等检查
    IdempotentRecord record = idempotentDao.findByToken(request.getIdempotentToken());
    if (record != null) {
        // 已经处理过，直接返回原结果
        return buildResult(record.getStatus(), record.getResultData());
    }

    // 2. 开启事务
    transactionTemplate.execute(() -> {
        // 3. 写入幂等表（唯一键保证不重复）
        idempotentDao.insert(new IdempotentRecord()
            .setToken(request.getIdempotentToken())
            .setStatus("PROCESSING")
            .setCreatedAt(now()));

        // 4. 执行业务：调用渠道、扣库存、创建订单...
        PayResult result = paymentService.processPayment(request);

        // 5. 更新幂等表状态
        idempotentDao.updateStatusAndResult(
            request.getIdempotentToken(),
            result.getStatus(),
            serialize(result)
        );
    });

    return payResult;
}
```

### 方案二：去重表（唯一索引）

**原理**：不额外申请Token，直接利用业务字段组合作为唯一键。常用的是"渠道编号 + 商户订单号"，因为这个组合在业务上天然唯一。

```sql
CREATE UNIQUE INDEX idx_channel_order
ON payment_transaction (channel_code, merchant_order_no);
```

```java
public PayResult handleCallback(ChannelCallback callback) {
    try {
        // 唯一索引冲突时自动报错，不会插入两条记录
        paymentTransactionDao.insert(new PaymentTransaction()
            .setChannelCode(callback.getChannel())
            .setMerchantOrderNo(callback.getOrderNo())
            .setChannelOrderNo(callback.getChannelOrderNo())
            .setAmount(callback.getAmount())
            .setStatus("SUCCESS")
            .setCallbackTime(now()));

        // 插入成功 → 首次回调，执行业务
        orderService.confirmOrder(callback.getOrderNo());

    } catch (DuplicateKeyException e) {
        // 唯一键冲突 → 重复回调，直接返回成功
        log.info("重复回调忽略: channel={}, orderNo={}",
            callback.getChannel(), callback.getOrderNo());
        return PayResult.success("重复回调已处理");
    }
}
```

**适用场景**：渠道回调处理、所有异步通知场景
**代价**：业务表和幂等逻辑耦合；唯一键设计要覆盖所有业务场景
**注意**：要区分"重复通知"和"部分成功"。重复通知返回成功是对的；但如果上一次回调处理失败了（状态是FAIL），新的成功回调应该触发重试，而不是静默忽略。

### 方案三：分布式锁 + 状态机

**原理**：对于状态变更类操作（如"待支付 → 已支付 → 已结算"），用分布式锁锁住订单ID，然后基于状态机的不可逆流转保证幂等。

```
状态机设计：
  INIT → PAYING → SUCCESS
            ↓
          FAIL
  SUCCESS → REFUNDING → REFUNDED

规则：
  - INIT 才能触发 PAYING
  - PAYING 才能触发 SUCCESS 或 FAIL
  - 状态已到 SUCCESS 的，不能再变成 FAIL
```

```java
public void transitionState(Long orderId, String targetState) {
    // 1. 加锁
    String lockKey = "order:state:" + orderId;
    distributedLock.tryLock(lockKey, 3, TimeUnit.SECONDS);

    try {
        Order order = orderDao.findById(orderId);

        // 2. 校验状态流转是否合法
        if (!isValidTransition(order.getStatus(), targetState)) {
            throw new IllegalStateException(
                "非法状态流转: " + order.getStatus() + " -> " + targetState);
        }

        // 3. 原子更新（乐观锁）
        int updated = orderDao.updateStatus(orderId, order.getStatus(), targetState);
        if (updated == 0) {
            throw new ConcurrencyException("并发状态更新冲突");
        }

        // 4. 触发后续动作
        afterStateChange(orderId, order.getStatus(), targetState);

    } finally {
        distributedLock.unlock(lockKey);
    }
}
```

**适用场景**：订单状态流转、退款状态流转、账户余额变动
**代价**：锁竞争带来性能损耗；状态机设计复杂，容易出现"漏网之鱼"
**核心优势**：即使请求乱序到达（支付成功回调先到，系统重启前的失败回调后到），状态机也能保证最终结果正确

---

### 三种方案横向对比

| 维度 | Token+幂等表 | 去重表 | 分布式锁+状态机 |
|------|-------------|--------|----------------|
| 实现复杂度 | 中 | 低 | 高 |
| 性能开销 | 多一次DB写入 | 依赖索引效率 | 锁竞争开销 |
| 业务侵入性 | 低 | 中 | 高 |
| 适用场景 | 同步下单 | 异步回调 | 状态变更 |
| 并发安全性 | 高 | 高 | 中 |
| 可维护性 | 好 | 最好 | 一般 |

**选型建议**：
- 渠道回调优先用去重表，最简单有效
- 用户下单用Token+幂等表，支持前端重试
- 核心账户操作（余额变动）用状态机，最严格

## 三、不同层级的幂等策略

很多工程师只在一个地方做了幂等处理，结果在其他层级还是出了问题。支付系统有三个层级，每个层级都要考虑。

### L1 网关层：通道方回调幂等

这是最容易出问题的层级。

微信支付、支付宝、银联的回调，都可能因为网络原因重发。一次支付成功，渠道可能给你发3-5次相同的回调。

```java
// 网关层处理回调的反模式
public void handleCallback(CallbackRequest callback) {
    // 错误做法：每次都处理
    paymentService.process(callback);

    // 正确做法：先查后处理
    if (hasProcessed(callback.getChannelOrderNo())) {
        return; // 已处理，直接返回
    }
    paymentService.process(callback);
}
```

**关键是"查"的效率**：不能用事务查，要用缓存（Redis）查。支付回调要求毫秒级响应，数据库查询太慢。

```java
// Redis缓存幂等标志
private static final String CALLBACK_PROCESSED_KEY = "cb:processed:%s:%s";

public boolean hasProcessed(String channel, String channelOrderNo) {
    String key = String.format(CALLBACK_PROCESSED_KEY, channel, channelOrderNo);
    return redisTemplate.hasKey(key);
}

public void markProcessed(String channel, String channelOrderNo) {
    String key = String.format(CALLBACK_PROCESSED_KEY, channel, channelOrderNo);
    // 设置1天过期，避免Redis膨胀
    redisTemplate.opsForValue().set(key, "1", 1, TimeUnit.DAYS);
}
```

---

### L2 应用层：商户订单号幂等

这个层级处理的是：同一个商户订单，不同用户操作。

用户多点了一次、前端按钮没禁用住、App闪退后重新打开...这些都会导致同一笔商户订单被重复提交。

**幂等Key的设计规范**：`商户ID + 订单号 + 操作类型`

```java
// 不要只用心单号，要加商户ID
// 因为不同的商户可能生成相同的订单号
String idempotentKey = merchantId + ":" + orderNo + ":" + actionType;
```

幂等表要定期归档。交易量大的系统，每天可能新增上亿条幂等记录。不归档，数据库会爆。

---

### L3 数据层：数据库唯一索引兜底

这是最后一道防线。

无论代码逻辑多严谨，生产环境总有意想不到的情况：代码Bug、运维操作失误、消息队列堆积...最终会有漏网之鱼。

数据库唯一索引就是"兜底"的：即使所有其他防线都失效，它也能保证不会有两条相同的记录。

```sql
-- 支付流水表，所有业务幂等的最后防线
CREATE TABLE payment_transaction (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    channel_code VARCHAR(32) NOT NULL COMMENT '渠道编号',
    merchant_id VARCHAR(32) NOT NULL COMMENT '商户编号',
    merchant_order_no VARCHAR(64) NOT NULL COMMENT '商户订单号',
    channel_order_no VARCHAR(128) COMMENT '渠道订单号',

    -- 关键：三重唯一键，兜住所有幂等场景
    UNIQUE KEY uk_channel_merchant_order (channel_code, merchant_id, merchant_order_no),

    amount DECIMAL(12,2) NOT NULL,
    status VARCHAR(16) NOT NULL,
    gmt_create DATETIME NOT NULL,
    gmt_modified DATETIME NOT NULL,

    INDEX idx_channel_order (channel_order_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 四、七大反直觉陷阱

### 陷阱1：幂等和事务不是一回事

这是新人最容易踩的坑。

幂等保证的是"多次请求，只处理一次"；事务保证的是"一组操作，要么全部成功，要么全部失败"。

两者解决的问题不同，手段也不同。

- 幂等：用唯一键、去重表、状态校验防止重复处理
- 事务：用数据库ACID特性保证原子性

一个典型错误：把幂等逻辑写在事务里，结果事务回滚了，幂等记录也清掉了，下次请求还是能进来。

```java
// 错误：事务和幂等混在一起
transactionTemplate.execute(() -> {
    idempotentDao.insert(token); // 事务内
    paymentService.process();
    // 如果process()抛异常，事务回滚，token记录也没了
});
```

**正确做法**：幂等写在事务外。先检查幂等，再开启事务。

---

### 陷阱2：幂等表本身也可能成为瓶颈

设计幂等表时，大多数人会忽略这个问题。

当TPS达到数万时，所有请求都要先写幂等表。幂等表变成了热点，高并发下锁竞争严重，反而拖累了整体性能。

**解决方案**：
1. 幂等表分库分表，按Token哈希分散
2. 批量异步写入：先写内存队列，后台批量落库
3. 用Redis替代数据库幂等表（纯内存，性能高一个数量级）

```java
// Redis实现的轻量级幂等（适用高并发场景）
public Result executeWithIdempotency(String key, Supplier<Result> action) {
    // SETNX保证原子性
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(key, "PROCESSING", 30, TimeUnit.SECONDS);

    if (!locked) {
        // 已被其他请求持有，检查状态
        String status = redisTemplate.opsForValue().get(key);
        if ("SUCCESS".equals(status)) {
            return getCachedResult(key);
        }
        throw new DuplicatedRequestException("请求正在处理中");
    }

    try {
        Result result = action.get();
        redisTemplate.opsForValue().set(key + ":result", serialize(result), 24, TimeUnit.HOURS);
        redisTemplate.opsForValue().set(key, "SUCCESS", 24, TimeUnit.HOURS);
        return result;
    } catch (Exception e) {
        redisTemplate.delete(key);
        throw e;
    }
}
```

### 陷阱3：第三方渠道的幂等Key不一定可靠

依赖方（微信/支付宝）提供的订单号作为幂等Key，看似稳妥，实际上有两个隐患：

**隐患1：不同环境订单号可能重复**

微信支付的订单号在开发环境和生产环境是独立的。测试时用生产订单号做幂等，可能覆盖真实交易。

**隐患2：部分渠道的超时重试会换订单号**

部分银行渠道超时后会生成新订单号重发，但实际上是同一笔交易。这种情况下，用渠道订单号做幂等就不够了，必须加上支付时间窗口（5分钟内相同金额相同卡号视为同一笔）。

### 陷阱4：幂等处理要区分"成功"和"处理中"

```java
public void handleCallback(CallbackRequest callback) {
    IdempotentRecord record = findRecord(callback);

    if (record == null) {
        // 首次处理
        doProcess(callback);
        return;
    }

    switch (record.getStatus()) {
        case "PROCESSING":
            // 正在处理中，可能是超时重试，不能再开新流程
            // 但要确认一下上一轮是否真的失败了
            checkAndRecovery(record.getOrderId());
            break;
        case "SUCCESS":
            // 已成功，直接返回
            break;
        case "FAIL":
            // 上次失败了，新回调应该触发重试，不是忽略
            doProcess(callback);
            break;
    }
}
```

### 陷阱5：补偿事务也要幂等

退款是支付的逆向操作。支付做了幂等，退款不做，后果更严重——用户可能重复退款，公司直接损失。

```java
public void doRefund(RefundRequest request) {
    // 退款也要幂等！用原支付单号+退款批次号做唯一键
    String idempotentKey = request.getPaymentId() + ":" + request.getRefundBatchNo();

    try {
        refundDao.insertIdempotent(idempotentKey); // 唯一键冲突则抛异常
    } catch (DuplicateKeyException e) {
        log.warn("重复退款请求忽略: paymentId={}", request.getPaymentId());
        return; // 已退过，直接返回
    }

    // 执行退款...
}
```

### 陷阱6：消息队列的消费幂等是另一套逻辑

支付系统内部用消息队列解耦（比如订单创建后发一条消息，触发库存扣减、积分发放等下游操作）。消息队列的消费幂等，与RPC调用的幂等有本质区别：

- RPC幂等：调用方感知失败，可以重试
- MQ消费幂等：MQ会无限重试，直到消费成功为止

因此，MQ消费幂等必须保证"无论消费多少次，结果都是对的"，而不是"消费一次就删掉消息"。

```java
// RocketMQ消费幂等的正确姿势
@RocketMQMessageListener(
    topic = "order_create",
    consumerGroup = "inventory_group",
    maxReconsumeTimes = 16  // RocketMQ默认重试16次后进入死信队列
)
public void onMessage(OrderCreateMessage message) {
    // 幂等检查：先查后写，是消费幂等的标准模式
    if (inventoryService.hasProcessed(message.getOrderId())) {
        return; // 已扣过，安全跳过
    }

    try {
        inventoryService.扣减库存(message.getOrderId(), message.getItems());
        inventoryService.markProcessed(message.getOrderId());
    } catch (Exception e) {
        // 扣库存失败，不要吃异常，要抛出去让MQ重试
        // 重要：RocketMQ同一个consumerGroup内，消息只会被一个实例消费
        // 重试时幂等检查会再次保护，不会重复扣
        throw e;
    }
}
```

### 陷阱7：历史数据迁移时的幂等漏洞

运维从备份恢复数据、手动修正数据、跨系统同步数据...这些操作往往会绕过应用层的幂等逻辑，直接操作数据库。

一旦脏数据进了幂等表/去重表，幂等机制就失效了。

**建议**：幂等表设计时加一个"数据来源"字段，区分"业务写入"和"数据修复"，并对数据修复做额外的幂等校验。

## 五、幂等 vs 可重入：两个完全不同的概念

这是所有新人都会混淆的一对概念，放在一起说清楚。

| 维度 | 幂等性（Idempotent） | 可重入（Reentrant） |
|------|---------------------|-------------------|
| 关注点 | 同一个请求，多次执行结果相同 | 同一个方法，被多个线程同时执行不出错 |
| 解决什么问题 | 重复请求、数据重复 | 并发安全、线程安全 |
| 实现手段 | 唯一键、状态机、去重表 | 锁、信号量、ThreadLocal |
| 作用域 | 系统级（全链路） | 方法级（单个函数） |
| 金融场景 | 必须（涉及钱） | 推荐（有并发就可能有问题） |

```java
// 可重入锁：同一个线程可以多次获取
private final ReentrantLock lock = new ReentrantLock();

public void doSomething() {
    lock.lock();
    try {
        // 这个方法可以被同一个线程递归调用，不会死锁
        doSomethingElse(); // 里面也调用了 doSomething()
    } finally {
        lock.unlock();
    }
}

// 幂等性：保证多次调用只执行一次
public synchronized void processOrder(Long orderId) {
    if (hasProcessed(orderId)) return; // 检查幂等
    // 业务逻辑...
    markProcessed(orderId);
}
```

**关键区别**：
- 可重入是线程安全概念，解决的是"多个线程同时调用"的问题
- 幂等是请求语义概念，解决的是"同一个请求被处理多次"的问题

一个系统可以同时是幂等的和可重入的，也可以只满足其中一个。支付系统要求两者兼备。

---

## 结语

幂等性是支付系统里少数几个"看似简单，实则处处是坑"的技术点。

它的复杂性来自于：网络不可靠、用户不可信、渠道不可控、系统总会出错。在这三个"不可"的夹击下，幂等性设计就成了保障资金安全的最后一道闸门。

本文的七个陷阱，是无数生产事故凝结出来的经验。没有一个是从书本上抄来的，都是实际踩过的坑。

**下期预告**：支付系统的阿喀琉斯之踵——分布式事务。幂等性保证的是"一次请求只处理一次"，但当一笔支付跨越多个系统（账户、积分、库存、渠道）时，如何保证所有系统要么一起成功，要么一起回滚？TCC、Saga、本地消息表，三种方案的选型决策树，一次讲清楚。
