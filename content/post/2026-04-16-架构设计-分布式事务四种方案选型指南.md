## 一个真实的投诉

用户打来电话："你们扣了我500块钱，但订单显示失败了，钱去哪儿了？"

查日志，还原了事情经过：

- 用户下单，选择微信支付
- 账户服务扣了500元余额，状态：SUCCESS
- 调用积分服务给用户加500积分时，网络超时
- 积分服务返回超时，账户服务回滚了扣款
- 但用户端微信支付已经成功，微信的回调也发了
- 最终：用户没收到积分，订单显示失败，但钱已经被微信收走了

这不是微信的问题，是系统没有正确处理分布式事务。

## 一、从酸碱平衡说起：CAP/BASE与支付场景

分布式事务的复杂度，来源于一个基本事实：**支付系统的不同模块，在不同的机器上，甚至在不同的公司里。**

单机数据库事务可以保证ACID——原子性、一致性、隔离性、持久性。但一旦跨越多个数据库、多个服务，ACID就失效了。

**CAP定理**说：一个分布式系统最多同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）中的两个。实际生产环境里网络分区一定会发生，所以实际上是在C和A之间做取舍。

**BASE理论**是对CAP中"强一致性"的妥协：
- **Basically Available**：基本可用，允许部分失败
- **Soft state**：软状态，允许数据暂时不一致
- **Eventually consistent**：最终一致性，数据经过一段时间后会达成一致

支付系统选择BASE，而不是强ACID。因为：
1. 支付不能长时间不可用（用户要能付款）
2. 允许短暂的数据不一致（事后对账可以修正）
3. 但最终必须正确（钱不能凭空消失）

理解了这个背景，才知道为什么没有"完美的分布式事务解决方案"——只有"适合当前业务场景的方案"。

## 二、四种方案概述

| 方案 | 原理 | 是否锁资源 | 业务侵入性 | 支付场景适用度 |
|------|------|-----------|-----------|--------------|
| 2PC/XA | 协调者统一 Prepare/Commit | 是，全局锁 | 低 | ★☆☆☆☆（渠道不支持）|
| TCC | Try-Confirm-Cancel 三阶段 | 否，预留资源 | 高 | ★★★★☆（账户类）|
| Saga | 长事务拆成多个短事务+补偿 | 否 | 中 | ★★★★★（跨系统长链路）|
| 本地消息表 | 本地表+消息队列，最终一致 | 否 | 低 | ★★★★☆（异步解耦）|

## 三、逐一拆解

### 3.1 两阶段提交（2PC）

#### 原理

```
第一阶段（Prepare）：
  协调者向所有参与者发送 Prepare 请求
  参与者执行事务（但不提交），写 redo/undo 日志
  参与者回复 YES/NO

第二阶段（Commit/Rollback）：
  所有参与者都回复 YES → 协调者发 Commit，所有人提交
  任何一个回复 NO → 协调者发 Rollback，所有人回滚
```

#### 支付场景为什么不选

2PC在支付系统几乎不可用，有三个致命原因：

**1. 协调者单点**：协调者挂了，所有参与者永远等 Commit/Rollback，资源被锁住

**2. 长时间锁表**：从 Prepare 到 Commit 之间，所有参与者都握着行锁/表锁。支付涉及微信/支付宝/银联，这些渠道方不可能让你锁他们的资源这么久

**3. 无法应对渠道故障**：微信支付超时后可能已经扣了钱，但2PC要求全局同时提交。微信不会配合你做两阶段提交

**结论**：2PC只适合"所有参与者都是你自己控制的数据库"的场景。支付系统有外部渠道，不适用。

### 3.2 TCC（Try-Confirm-Cancel）

#### 原理

TCC把每个业务操作拆成三个阶段：

```
Try：预留资源（冻结/锁定，但不真正扣减）
Confirm：确认使用（真正扣减冻结的资源）
Cancel：取消释放（释放冻结的资源）
```

#### 实战：账户扣款 + 积分发放

```java
// 账户服务：扣款 TCC
@LocalTCC
public class AccountTccService {

    @TryMethod
    public void tryDeduct(Long userId, BigDecimal amount) {
        // 1. 检查余额是否足够
        Account account = accountDao.findByUserId(userId);
        if (account.getBalance().compareTo(amount) < 0) {
            throw new BizException("余额不足");
        }

        // 2. 冻结金额（余额不变，专门记一笔冻结记录）
        frozenAccountDao.insert(new FrozenRecord()
            .setUserId(userId)
            .setAmount(amount)
            .setStatus("FROZEN")
            .setExpireTime(now().plusMinutes(30))); // 30分钟超时自动解冻

        // 3. 记录 TCC 事务上下文（供 Confirm/Cancel 使用）
        TransactionContext.set(new TccContext(userId, amount, "DEDUCT"));
    }

    @ConfirmMethod
    public void confirmDeduct(TransactionContext ctx) {
        // 真正扣减余额，减少冻结记录
        accountDao.deductBalance(ctx.getUserId(), ctx.getAmount());
        frozenAccountDao.release(ctx.getUserId(), ctx.getAmount());
    }

    @CancelMethod
    public void cancelDeduct(TransactionContext ctx) {
        // 释放冻结金额，恢复可用余额
        frozenAccountDao.unfreeze(ctx.getUserId(), ctx.getAmount());
    }
}

// 积分服务：加积分 TCC
@LocalTCC
public class PointsTccService {

    @TryMethod
    public void tryAddPoints(Long userId, Integer points) {
        // 积分相对简单，只需要记录要加多少
        TransactionContext.set(new TccContext(userId, points, "ADD_POINTS"));
    }

    @ConfirmMethod
    public void confirmAddPoints(TransactionContext ctx) {
        pointsDao.addPoints(ctx.getUserId(), ctx.getPoints());
    }

    @CancelMethod
    public void cancelAddPoints(TransactionContext ctx) {
        // 积分加错了要减回来（实际业务中可能不允许反向操作，这里仅做示例）
        pointsDao.deductPoints(ctx.getUserId(), ctx.getPoints());
    }
}
```

#### TCC的三个致命陷阱

TCC看起来很美，但有三个坑，几乎每个第一次用TCC的团队都会踩。

**陷阱1：空回滚**

场景：Try请求超时，Confirm/Cancel先执行了，但Try从未真正执行过。

```java
// 错误做法：Cancel 时直接释放资源
@CancelMethod
public void cancelDeduct(TransactionContext ctx) {
    frozenAccountDao.unfreeze(ctx.getUserId(), ctx.getAmount());
    // 问题：如果 Try 没执行过，这里会多释放一次冻结
}

// 正确做法：Cancel 前先检查 Try 是否执行过
@CancelMethod
public void cancelDeduct(TransactionContext ctx) {
    // 通过分布式事务 ID 查询 Try 是否已执行
    TccTransactionLog log = tccLogDao.findByXid(ctx.getXid());
    if (log == null || !"TRY_SUCCESS".equals(log.getStatus())) {
        // Try 从未执行，直接返回，不做任何操作
        return;
    }
    frozenAccountDao.unfreeze(ctx.getUserId(), ctx.getAmount());
}
```

**陷阱2：悬挂**

场景：Try 执行了但失败了，Cancel 还没来得及执行，网络恢复后超时导致 Confirm 请求先到了。此时资源已经被永久冻结。

```java
// 正确做法：Confirm/Cancel 执行前，再次校验 Try 是否真正成功
@ConfirmMethod
public void confirmDeduct(TransactionContext ctx) {
    TccTransactionLog log = tccLogDao.findByXid(ctx.getXid());
    if (log == null || !"TRY_SUCCESS".equals(log.getStatus())) {
        // Try 未成功，Confirm 不应该执行
        throw new IllegalStateException("Try phase not completed, cannot confirm");
    }
    accountDao.deductBalance(ctx.getUserId(), ctx.getAmount());
    frozenAccountDao.release(ctx.getUserId(), ctx.getAmount());
}
```

**陷阱3：幂等**

TCC的每个阶段都可能因为网络原因被重复调用。Confirm和Cancel必须自己保证幂等。

```java
@ConfirmMethod
public void confirmDeduct(TransactionContext ctx) {
    // 必须先检查是否已经 Confirm 过
    TccTransactionLog log = tccLogDao.findByXid(ctx.getXid());
    if ("CONFIRMED".equals(log.getStatus())) {
        return; // 重复调用，安全忽略
    }
    accountDao.deductBalance(ctx.getUserId(), ctx.getAmount());
    tccLogDao.updateStatus(ctx.getXid(), "CONFIRMED");
}
```

#### TCC适用场景

- 账户余额变动（可以用冻结作为 Try 资源）
- 优惠券锁定/核销
- 库存预占/释放

**不适用**：外部渠道调用（微信支付不能配合你做 TCC）

### 3.3 Saga模式

#### 原理

Saga把一个长事务拆成多个短事务，每个短事务都有对应的补偿操作：

```
正常执行：A → B → C → D（每步成功都记录）
出错了：  A → B → [C失败] → 补偿B → 补偿A（回滚）
```

有两种实现方式：

- **Choreography（编排式）**：各服务通过事件驱动，下游服务订阅上游事件，自己判断是否触发补偿
- **Orchestration（编排器式）**：用一个中心化的 Saga Orchestrator 统一管理所有参与者的执行和补偿

#### 实战：跨境汇款（Orchestration模式）

```java
// Saga Orchestrator：管理整个跨境汇款流程
public class RemittanceSaga {

    public void execute(RemittanceRequest request) {
        SagaContext ctx = new SagaContext();
        ctx.setXid(UUID.randomUUID().toString());
        ctx.setRequest(request);

        try {
            // Step 1: 扣用户本币账户
            accountService.deduct(request.getUserId(), request.getAmount());
            ctx.logStep("DEDUCT", "SUCCESS");

            // Step 2: 外汇换汇
            String exchangeOrder = forexService.convert(
                request.getAmount(), request.getFromCurrency(), request.getToCurrency());
            ctx.logStep("FOREX", "SUCCESS", exchangeOrder);

            // Step 3: 汇入收款人账户
            beneficiaryService.credit(request.getBeneficiaryId(), exchangeOrder);
            ctx.logStep("CREDIT", "SUCCESS");

        } catch (Exception e) {
            // 补偿：从最后一步往前补偿
            compensate(ctx, e);
        }
    }

    private void compensate(SagaContext ctx, Exception cause) {
        // 补偿严格按照倒序执行
        for (StepLog step : ctx.getStepsReversed()) {
            try {
                switch (step.getName()) {
                    case "CREDIT":
                        // 扣回收款人账户
                        beneficiaryService.debit(ctx.getRequest().getBeneficiaryId(), step.getRef());
                        break;
                    case "FOREX":
                        // 换汇回滚（如果汇率服务支持）
                        forexService.rollbackConvert(step.getRef());
                        break;
                    case "DEDUCT":
                        // 返还用户账户（注意：不能简单加回去，要有幂等保护）
                        accountService.refund(ctx.getRequest().getUserId(), ctx.getRequest().getAmount());
                        break;
                }
                step.setCompensated(true);
            } catch (Exception e) {
                // 补偿失败，记录并告警，等待人工介入
                alertService.alert("Saga补偿失败: xid=" + ctx.getXid() + ", step=" + step.getName());
            }
        }
    }
}
```

#### Saga vs TCC：核心区别

| 维度 | TCC | Saga |
|------|-----|------|
| 资源锁定 | Try阶段预留，不锁真实资源 | 不锁定资源，快速释放 |
| 补偿方式 | 逆操作（Cancel） | 正向补偿（Compensate），不是简单逆操作 |
| 适用场景 | 资源预留类操作（账户、库存） | 长链路跨系统（汇款、订单履约） |
| 事务时长 | 短（秒级） | 可长（分钟级甚至更长） |
| 实现复杂度 | 高（每个接口写3个方法） | 中（补偿逻辑需要单独设计） |

**一个关键区别**：TCC的Cancel是"撤销Try的操作"，而Saga的Compensate是"做一笔新的反向交易来对冲"。Saga的补偿不一定是严格的逆操作（比如汇率变了，补偿不能简单地把钱加回去，要考虑汇率差异）。

### 3.4 本地消息表（事务性消息）

#### 原理

把业务表和消息表放在同一个数据库事务里，通过本地事务保证"业务和消息要么同时成功，要么同时失败"，然后通过定时轮询或消息队列的Confirm机制把消息发出去。

```
1. 开启本地事务
2. 插入业务数据（订单表）
3. 插入消息表（待发送的消息）
4. 提交本地事务（业务和消息同时落库）
5. 后台定时任务轮询消息表，发送消息
6. 消费端处理后，ACK → 更新消息状态为已发送
```

#### 实战：订单创建后触发库存扣减

```java
// 本地消息表
CREATE TABLE outbox (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    aggregate_type VARCHAR(64) NOT NULL,  -- 聚合类型，如 "ORDER"
    aggregate_id VARCHAR(64) NOT NULL,     -- 聚合 ID
    event_type VARCHAR(64) NOT NULL,       -- 事件类型，如 "ORDER_CREATED"
    payload TEXT NOT NULL,                 -- 消息内容（JSON）
    status VARCHAR(16) NOT NULL DEFAULT 'PENDING',  -- PENDING/SENT/FAILED
    retry_count INT DEFAULT 0,
    gmt_create DATETIME NOT NULL,
    gmt_modified DATETIME NOT NULL,
    UNIQUE KEY uk_aggregate_event (aggregate_type, aggregate_id, event_type)
) ENGINE=InnoDB;

// 订单服务：创建订单时写入消息表
@Transactional
public Order createOrder(OrderCreateRequest request) {
    // 1. 业务处理
    Order order = new Order()
        .setUserId(request.getUserId())
        .setAmount(request.getAmount())
        .setStatus("CREATED");
    orderDao.insert(order);

    // 2. 写入消息表（在同一事务内）
    outboxDao.insert(new Outbox()
        .setAggregateType("ORDER")
        .setAggregateId(order.getId().toString())
        .setEventType("ORDER_CREATED")
        .setPayload(JSON.toJSONString(new OrderCreatedEvent(order)))
        .setStatus("PENDING"));

    // 3. 事务提交时，业务和消息同时落库
    // 如果业务回滚，消息也不会留下（同一事务）
    return order;
}

// 后台轮询任务：发送消息
@Scheduled(fixedDelay = 1000)
public void pollOutbox() {
    List<Outbox> messages = outboxDao.findPending(100); // 每次取100条
    for (Outbox msg : messages) {
        try {
            // 发送到 MQ
            mqProducer.send(msg.getTopic(), msg.getPayload());

            // 更新状态为已发送
            outboxDao.updateStatus(msg.getId(), "SENT");

        } catch (Exception e) {
            // 发送失败，重试计数
            outboxDao.incrementRetry(msg.getId());
            if (msg.getRetryCount() >= 3) {
                outboxDao.updateStatus(msg.getId(), "FAILED");
                alertService.alert("消息发送失败超过3次: " + msg.getId());
            }
        }
    }
}
```

#### 进化：RocketMQ事务消息

手动轮询消息表太原始。RocketMQ原生支持事务消息，原理类似但实现更优雅：

```
1. 发送半消息（对消费端不可见）
2. 执行本地事务
3. 本地事务成功 → 提交半消息，消费端可见
   本地事务失败 → 回滚半消息，消费端永远看不到
4. 如果 RocketMQ 长时间没收到事务结果（即"回查超时"），
   会主动向发送方查询："这笔事务到底成功了没有？"
   发送方根据本地事务表返回 COMMIT 或 ROLLBACK
```

**关键理解**：MQ是"被动回查方"，发送方是"被回查方"。MQ不会主动回调发送方，而是等待发送方通知。如果发送方超时没响应，MQ会主动发起回查请求。

```java
// RocketMQ 事务消息示例
@Transactional
public void createOrderWithTransactionalMq(OrderCreateRequest request) {
    Order order = orderDao.insert(new Order()
        .setUserId(request.getUserId())
        .setAmount(request.getAmount()));

    // 发送半消息，RocketMQ 暂不投递给消费者
    kafkaMqProducer.sendInTransaction("order_topic",
        JSON.toJSONString(new OrderCreatedEvent(order)),
        (msg, arg) -> {
            // 本地事务回调：检查订单是否真的创建了
            Order saved = orderDao.findById(order.getId());
            return saved != null ? LocalTransactionState.COMMIT_MESSAGE : LocalTransactionState.ROLLBACK_MESSAGE;
        }, null);
}
```

#### 本地消息表 vs TCC

| 维度 | 本地消息表 | TCC |
|------|-----------|-----|
| 一致性 | 最终一致（有延迟） | 强一致（实时） |
| 实时性 | 准实时（秒级延迟） | 实时 |
| 实现复杂度 | 低 | 高 |
| 适用场景 | 可接受延迟的异步场景 | 必须实时一致的账户操作 |

**选型判断**：库存扣减可以用本地消息表（用户可以接受等几秒），但账户余额变动必须用TCC（余额必须实时准确）。

## 四、选型决策树

这是全文最有价值的部分，建议收藏。

```flow
st=>start: 收到跨系统操作请求
cond1=>condition: 涉及外部渠道？
op_ext=>operation: 本地消息表 + 渠道异步回调确认
cond2=>condition: 涉及几个内部服务？
op_realtime=>condition: 有账户/余额等金融操作？
op_tcc=>operation: TCC（Try预留/Confirm/Cancel）
op_saga=>operation: Saga（Orchestration编排）
op_msg=>operation: 本地消息表（最终一致即可）
e=>end: 完成

st->cond1
cond1(yes)->op_ext->e
cond1(no)->cond2
cond2(yes)->op_realtime
cond2(no)->op_msg->e
op_realtime(yes)->op_tcc->e
op_realtime(no)->op_saga->e
```

**重新理解分类维度**：决策树的第一维度是"是否涉及外部渠道"（渠道不配合你做TCC/Saga），第二维度是"内部系统的复杂度"。涉及渠道时，本地消息表是唯一选择；不涉及渠道时，再看实时性要求（金融操作→TCC，非金融长链路→Saga，可接受延迟→本地消息表）。

**一句话总结**：
- 渠道回调 → 本地消息表（渠道不配合你做TCC）
- 账户/余额/积分 → TCC（必须强实时）
- 长链路多系统（汇款/履约） → Saga（不锁资源，可补偿）
- 可接受延迟的异步操作 → 本地消息表
- 可接受延迟的异步操作 → 本地消息表

## 五、三个真实踩坑案例

### 案例1：TCC空回滚导致余额多出

某团队用了TCC做账户扣款，Try阶段冻结金额，Confirm阶段扣减冻结金额并释放。测试没问题，上线后出现用户余额凭空多了几万。

原因：网络抖动导致Try超时，Cancel被执行了。但Cancel里写了"冻结金额-0.5"，而Try根本没执行成功，冻结记录不存在。这行SQL执行了但没影响任何行（0 row affected），没人注意到。

修复：Cancel执行前必须检查Try是否真正执行成功（通过事务日志或状态标记）。

### 案例2：Saga补偿遗漏导致资金损失

某跨境汇款业务用了Saga，换汇成功后，收款人账户服务超时了。系统记录了"换汇成功-收款失败"，然后补偿了换汇。但换汇的补偿是"换回原币种"，汇率已经变了——用户实际损失了汇率差价。

问题：补偿逻辑设计时没有考虑汇率变动的场景。补偿不是简单的逆操作。

修复：Saga的补偿逻辑需要作为"一等公民"来设计，不能简单写成"逆向操作"。对于汇率敏感的补偿，要记录补偿时的汇率或使用更复杂的对冲机制。

### 案例3：本地消息表消息丢失

某团队用本地消息表做订单创建后的库存扣减通知。凌晨2点数据库主从切换，有1分钟写入的消息没有同步到从库，从库里的消息表是空的。这1分钟创建的订单，库存扣减通知全部丢失。

问题：本地消息表依赖数据库复制，但如果用的是异步复制（半同步复制只保证一个从库收到），主从切换时可能丢消息。

修复：用半同步复制确保消息至少写到一个从库，或者在应用层做双写（主库+从库都写消息表，查询时取任一个有的）。

## 六、最佳实践总结

**1. 不要迷信强一致**

支付系统的很多场景，最终一致性就够了。对账系统会在T+1发现并修正不一致。强一致意味着更复杂的实现和更低的可用性。

**2. 补偿逻辑要作为一等公民设计**

TCC的Cancel和Saga的Compensate不是"事后打补丁"，而是要在设计阶段就认真思考的逻辑。补偿失败怎么办？补偿时汇率变了怎么办？补偿时原数据已经被修改了怎么办？

**3. 事务框架是工具，业务设计才是根本**

Seata、Apache ServiceComb只是帮你降低了实现成本，但业务逻辑设计不对，用什么框架都会出问题。

**4. 监控和告警是最后一道防线**

无论用哪种方案，总会有补偿失败、消息丢失、状态不一致的情况。完善的监控和告警能在第一时间发现这些问题，而不是等用户投诉。
