## 背景：一个 off-by-one 错误引发的思考

你向 AI 编程工具描述一个问题：函数里有个简单的 off-by-one 错误，把 `range(len(x) - 1)` 改成 `range(len(x))` 就好了。

AI 确实修好了这个 bug。但代价是：

- 整个函数被重写了
- 原本的变量名被改成了「更规范」的名称
- 凭空多出了 `None` 检查、`np.asarray` 转换、有限值掩码
- 原有的绘图逻辑整个被替换
- diff 行数是预期修改的 **30 倍**

**这就是 Over-editing 问题。** AI 模型在修复代码时，总是超出必要范围地改写更多内容。输出在功能上正确，但结构上与原始代码的差异远超最小修复所需。

这比你想象的更严重。代码评审已经是开发流程的瓶颈——评审人需要理解「改了什么、为什么改、是否安全」。一个把整个函数重写一遍的模型，哪怕结果正确，也极大增加了评审负担。更隐蔽的是：**这类过度修改对测试套件完全不可见**。代码质量在一次又一次的「顺手优化」中悄悄下降，而测试依然全部通过。

## 量化 Over-editing：两个核心指标

要研究一个问题，首先要能度量它。作者没有用 LLM 来注入 bug（这正是大多数现有基准的做法），而是从 BigCodeBench 中程序化地腐化 400 个问题——翻转一个比较运算符（`<` → `<=`）、把 `+` 换成 `-`、把布尔值从 `True` 改成 `False` 等。

每个腐化后的样本在语法上仍然有效，且经验证会破坏对应的测试用例。这意味着**正确答案就是腐化的反向操作**，从而在构造上保证了最小编辑。

### 指标一：Token 级 Levenshtein 距离

标准 Levenshtein 距离计算字符级别的最小插入、删除和替换次数。作者使用 Python tokenizer 将代码切分为原子语法单元（`def`、`add`、`(`、`a`、`,`、`b`、`)`、`:`、`return`……），再在 token 序列而非原始字符上计算 Levenshtein 距离。这样 `def add(a, b):` 和 `def someotherfunctionname(a, b):` 在字符层面距离为 19，但在 token 层面距离仅为 1。

设 $C$ 为腐化后的代码，$G$ 为正确答案，$M$ 为模型输出。真正的最小编辑是 $D_{\text{true}} = d(G, C)$，模型的编辑是 $D_{\text{model}} = d(M, C)$，得相对编辑得分：

$$S(M) = D_{\text{model}} - D_{\text{true}}$$

**分值越接近零，说明模型越接近最小编辑。**

### 指标二：Added Cognitive Complexity（认知复杂度增量）

认知复杂度衡量代码的可理解难度，对嵌套、递归、混合逻辑运算符和非线性控制流进行惩罚。由于所有腐化操作只改变值而不改变结构，正确的修复不应该增加任何认知复杂度。模型输出中任何认知复杂度的增加都是无意义的。

> `process` 函数的认知复杂度为 6（嵌套两层 if 分别 +1、+2、+3）

## 各模型 Over-editing 实测：差距悬殊

用上述方法对主流前沿模型进行评测，结果如下（仅展示推理模式，完整表格见原文）：

| 模型 | Pass@1 ↑ | 归一化 Levenshtein ↓ | Added CC ↓ |
|------|---------|---------------------|------------|
| **Claude Opus 4.6** | **0.912** | **0.060** | **0.200** |
| GLM-5 High | 0.859 | 0.099 | 0.320 |
| Gemini 3.1 Pro Preview | 0.858 | 0.145 | 0.501 |
| Qwen 3.6 Plus | 0.858 | 0.145 | 0.048 |
| Kimi 2.5 | 0.835 | 0.151 | 0.770 |
| DeepSeek R1 | 0.820 | 0.232 | 0.673 |
| **GPT-5.4** | 0.723 | **0.395** | **2.313** |

**GPT-5.4 是在推理模式下 Over-editing 最严重的模型**，归一化 Levenshtein 距离高达 0.395（Claude Opus 4.6 的 6.6 倍），认知复杂度增加了 2.313（最低的 Qwen 3.6 Plus 仅为 0.048）。然而更值得关注的是，GPT-5.4 在正确率上也是最弱的之一，Pass@1 只有 0.723。

**Claude Opus 4.6 是全场最佳**——正确率最高（0.912），同时 Levenshtein 距离最低（0.060），认知复杂度增加也最少（0.200）。

### 推理模型真的「想太多」吗？

推理模型通常被认为在编程任务上更强，这个假设在正确率上成立。但如果只看编辑风格，结论就反转了。

将推理模型与非推理模型配对比较（来自同一模型家族），在通用设置（未给出显式指令）下，**大多数推理模型的编辑距离都高于其非推理版本**：

- GPT-5.4 推理版 Levenshtein = 0.395，非推理版 = 0.327
- DeepSeek R1 推理版 Levenshtein = 0.232，非推理版 = 0.232（持平）
- **Claude Opus 4.6 是唯一例外**——推理版（0.060）显著低于非推理版（0.079）

推理过程给了模型更多空间去「优化」本来不需要改动的代码，形成了「想太多 → 改太多」的模式。

## 提示词约束：简单一招就能改善

在提示词中加入一句：

> **"IMPORTANT: Try to preserve the original code and the logic of the original code as much as possible."**

效果出乎意料地好。**所有模型的 Levenshtein 距离都下降了**，而且推理模型的改善幅度远大于非推理模型——它们的指令跟随能力更强，约束效果也更显著。

这说明了一个重要事实：**Over-editing 在很大程度上不是能力问题，而是默认行为问题。** 推理模型尤其容易被「引导」到更克制的编辑风格上，一旦给出明确约束，立刻就能看到改善。

## 如何训练更「克制」的模型？

### 实验设置

以 **Qwen3 4B 2507 Instruct** 为基础模型，用 DeepCoder 创建程序化腐化的合成数据集，同时用基础模型自身通过 self-distillation 生成合成数据集（每问题生成 8 个补全，过滤出功能正确且 Levenshtein 距离最低的样本）。

对比四种微调方法：

| 方法 | 说明 |
|------|------|
| **SFT** | 在程序化数据集上直接监督微调 |
| **rSFT** | 拒绝采样 SFT：在 self-distillation 数据集上只保留每个样本 Levenshtein 最低的 3 个补全 |
| **DPO** | 偏好优化：最高与最低 Levenshtein 样本两两配对 |
| **RL** | 强化学习：奖励结合功能正确性和 Levenshtein 编辑最小性 |

RL 的奖励结构：
```python
r = r_edit + 0.1   # 生成通过所有测试用例
r = -0.2            # 否则（编辑对了但破坏功能）
```

### 域内结果：SFT 看似完美

| 模型 | Pass@1 ↑ | 归一化 Levenshtein ↓ | Added CC ↓ |
|------|---------|---------------------|------------|
| Baseline (0-shot) | 0.735 | 0.169 | 0.731 |
| Baseline (8-shot) | 0.775 | 0.115 | 0.479 |
| **SFT** | **0.932** | **0.002** | **0.000** |
| rSFT | 0.782 | 0.100 | 0.435 |
| DPO | 0.752 | 0.021 | 0.113 |
| RL | 0.802 | 0.046 | 0.112 |

SFT 在域内数据上的表现好得不可思议——Levenshtein 几乎为零，认知复杂度完全不变。这立刻引发了作者警觉：**模型是不是只是在记忆特定腐化的反向操作，而不是真的学会了最小编辑？**

### 域外泛化：SFT 完全失效

用与训练集完全不同的腐化类型构建测试集，重新评估：

| 模型 | Pass@1 ↑ | 归一化 Levenshtein ↓ | Added CC ↓ | LiveCodeBench Δ |
|------|---------|---------------------|------------|-----------------|
| SFT | **0.458** | **-0.008** | **0.006** | **-0.149** |
| rSFT | 0.780 | 0.107 | 0.501 | -0.069 |
| DPO | 0.787 | 0.092 | 0.348 | -0.046 |
| **RL** | 0.782 | **0.050** | **0.185** | **+0.006** |

SFT 在域外**彻底崩溃**——Pass@1 跌到 0.458（低于基线的 0.735），甚至无法正确修复 bug。它只是记住了训练集中的特定腐化反转模式，对新类型的改动完全无效。

**RL 是唯一真正泛化的方法**，在 Levenshtein 和认知复杂度上全面优于基线，而且 LiveCodeBench 性能完全没有退化（Δ = +0.006）。这与更广泛的共识一致：**SFT 在记忆，RL 在泛化。**

### LoRA 是否够用？

由于这个任务本质上是调整风格而非学习新知识，探索 LoRA 是否足够：

| Rank | Pass@1 ↑ | 归一化 Levenshtein ↓ | Added CC ↓ | LiveCodeBench Δ |
|------|---------|---------------------|------------|-----------------|
| 1 | 0.738 | 0.166 | 0.676 | -0.022 |
| 8 | 0.775 | 0.112 | 0.426 | -0.022 |
| 16 | 0.805 | 0.087 | 0.328 | -0.005 |
| 32 | 0.795 | 0.065 | 0.235 | -0.011 |
| 64 | 0.797 | 0.051 | 0.160 | +0.001 |
| Full RL | 0.782 | 0.050 | 0.185 | +0.006 |

**Rank 64 的 LoRA 在 Levenshtein 距离上几乎追平全量 RL**，在认知复杂度上甚至更优。基础能力已具备的情况下，少量额外参数就足以迁移编辑风格，全量微调在大多数场景下并非必要。

## 总结：Over-editing 是可测量、可改善的问题

Over-editing 的核心张力在于：**AI 编程工具越强，开发者越信任它，它改的代码就越多；而改得越多，代码库的熵增就越难被察觉。**

本文揭示了几个关键结论：

1. **GPT-5.4 是 Over-editing 重灾区**，且正确率也最低；Claude Opus 4.6 是克制的典范
2. **推理模型普遍比非推理模型更容易 Over-editing**，但对约束的响应也更强烈
3. **一句简单的提示词就能显著改善**所有模型的编辑最小性
4. **RL 是唯一能泛化的训练方法**，SFT 存在严重的灾难性遗忘，LoRA（rank 64）是性价比最高的生产方案

对于实际使用，最直接的收益是：**向你的 AI 编程工具明确传达「保持原样」的意图。** 在提示词中加入 minimal edit 指令，能在大多数场景下立刻减少不必要的代码改动。如果要在 pipeline 中内置优化，RL + LoRA 是最务实的路径——低 rank 即可见效，且几乎不影响基线编码能力。

Over-editing 不是能力不足，而是默认行为。用对约束，这个问题是可以在当下就被改善的。
