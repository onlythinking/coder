## 一次被骂了一小时的支付故障

某天中午，客服接到大量投诉：用户支付一直失败，点击支付后转圈圈，最后提示"支付失败"。

查监控：通道全都在正常运行，但成功率从95%掉到了12%。

根因查到了：财务上周调整了通道费率，把另一个通道的费率降低了0.1%。路由系统重新计算综合得分后，流量自动倾斜到了那个降了费率的通道——但这个通道对部分用户群体的成功率只有10%（因为那个通道的风控策略对某省手机号段特别严格），大量用户被风控拦截了。

这就是支付路由失控的典型案例：没有基于真实成功率选通道，只看费率，不看综合转化率。

---

## 一、什么是支付路由

支付路由：用户在发起支付时，系统根据用户信息、金额、通道状态等因素，从多个可用支付通道中选出最优的那个。

"最优"不是固定答案，不同场景下的最优通道不一样：

- 金额小（100元以下）：优先选费率最低的通道，失败代价小
- 金额大（5000元以上）：优先选成功率最高的通道，失败一次客诉成本高
- 新用户首单：优先选成功率最高的通道，宁可多付费率也要保证成功
- 高价值用户：优先选体验最好的通道（成功率高、不弹窗），客单价值高值得投入

**路由要解决的核心问题**：在通道成本（费率）和通道收益（成功率×金额）之间，找到综合收益最高的选择。

## 二、路由决策的五维度

每笔支付请求，系统要综合评估五个维度：

| 维度 | 说明 | 对路由的影响 |
|------|------|------------|
| **通道费率** | 通道收取的手续费比例 | 费率越低，成本越低 |
| **通道成功率** | 该通道对该用户的真实成功率 | 成功率越高，转化损失越小 |
| **单笔限额** | 通道的单笔/每日限额 | 超过限额的通道不可用 |
| **用户画像** | 用户的历史支付习惯、设备、地区 | 决定哪个通道成功率高 |
| **通道状态** | 通道是否在维护、是否超额 | 不可用时必须切换 |

### 2.1 通道成功率是动态的，不是静态的

大多数人对通道成功率的认知是错的：以为"微信通道成功率是95%"。

实际上，微信通道对不同用户群体的成功率差异巨大：

- 对广东用户成功率：97%
- 对某省某号段用户成功率：41%（因为那个号段被风控，某渠道实测数据，仅供示例）
- 对iOS用户成功率：96%
- 对Android特定ROM用户成功率：78%（因为微信在部分ROM上有兼容问题）

所以路由决策里的"成功率"应该是：**该通道对当前这笔交易特定用户的预测成功率**，而不是通道的平均成功率。

> **说明**：以上五个维度是路由决策的"输入参数"，但具体怎么用这些参数做决策，是下一节要讲的路由策略——规则路由靠人工配置参数规则，权重路由靠量化模型打分，智能路由靠机器学习自动学习参数。五维度和三种策略是不同层次的概念，不是对应关系。

## 三、三种路由策略演进

### 策略一：规则路由（Rule-Based）

最早的路由方式，靠人工配置规则。

```java
public Channel selectChannel_RuleBased(PaymentRequest request) {
    // 规则1：大额走成功率最高的通道
    if (request.getAmount().compareTo(new BigDecimal("5000")) > 0) {
        return channelService.getByCode("wechat_high_success");
    }

    // 规则2：特定地区走特定通道（某省用户微信风控严，换支付宝）
    if (addressService.isHighRiskProvince(request.getUserProvince())) {
        return channelService.getByCode("alipay");
    }

    // 规则3：新用户走推荐通道（初期和通道方有联合运营活动）
    if (userService.isNewUser(request.getUserId())) {
        return channelService.getByCode("wechat_new_user_promo");
    }

    // 默认走最低费率通道
    return channelService.getLowestRateChannel();
}
```

**优点**：简单，运营可干预，出问题容易追溯。
**缺点**：规则越来越多之后变成一团乱麻，维护成本高；无法适应实时变化的通道质量。

---

### 策略二：权重路由（Weight-Based）

给每个通道打一个综合得分（成功率、费率、限额等因素加权），按得分分配流量比例。

```java
public Channel selectChannel_WeightBased(PaymentRequest request) {
    List<ChannelScore> candidates = channelService.getAvailableChannels().stream()
        .map(ch -> {
            // 成功率得分（成功率越高分数越高，权重最大）
            double successScore = ch.getSuccessRate(request) * 100 * 0.5;

            // 费率得分（费率越低分数越高，换算成百分制）
            // 注意：0.006（即0.6%）为示例基准费率，实际业务中应取当前最低可用通道费率或动态基准
            double rateScore = Math.max(0, (1 - ch.getRate() / 0.006)) * 100 * 0.3;

            // 限额得分（剩余额度越充足分数越高）
            double quotaScore = ch.getRemainingQuota(request.getDate())
                .divide(ch.getDailyQuota(), 4, RoundingMode.HALF_UP) * 100 * 0.2;

            double totalScore = successScore + rateScore + quotaScore;
            return new ChannelScore(ch, totalScore);
        })
        .filter(cs -> cs.score > 0)  // 过滤不可用通道
        .collect(Collectors.toList());

    // 按得分加权随机选择
    return weightedRandomSelect(candidates);
}
```

**核心公式**：
```
综合得分 = 成功率 × 权重1 + (1 - 费率/基准费率) × 权重2 + 剩余额度比例 × 权重3
```

**权重的调优**：权重的选择直接影响路由效果。可以通过A/B测试找到最优权重配比。不同业务阶段权重重心不同：业务早期追求高成功率（成功率权重高），稳定期追求成本最优（费率权重高）。具体数值因业务而异，代码里的0.5/0.3/0.2仅供参考，实际需按自身数据调优。

### 策略三：智能路由（ML-Based）

当通道数量超过5个、用户量超过一定规模时，基于规则的权重配置就无法应对复杂的通道特性组合了。

```python
# 智能路由：基于历史数据训练通道选择模型
def select_channel_ml(request: PaymentRequest) -> Channel:
    # 1. 构造特征向量
    features = {
        'amount': request.amount,
        'user_age_days': user_service.get_user_age_days(request.user_id),
        'user_transaction_count': user_service.get_transaction_count(request.user_id),
        'device_type': request.device_type,  # iOS/Android/Web
        'province': request.province,
        'hour_of_day': request.timestamp.hour,
        'channel_success_rate_7d': channel.get_7d_success_rate(request),
        'channel_rate': channel.rate,
        'channel_quota_remaining_pct': channel.quota_remaining_pct(),
    }

    # 2. 用模型预测每个通道的成功概率
    predictions = {}
    for ch in available_channels:
        # 模型输入：特征 + 通道历史成功率
        model_input = {**features, 'channel_id': ch.id}
        predictions[ch.id] = model.predict_success_rate(model_input)

    # 3. 综合得分 = 预测成功率 × 金额 - 通道手续费
    # 期望收益 = 成功概率 × (交易金额 - 手续费)
    channel_returns = []
    for ch in available_channels:
        expected_return = predictions[ch.id] * request.amount * (1 - ch.rate)
        channel_returns.append((ch, expected_return))

    # 4. 选期望收益最高的通道
    return max(channel_returns, key=lambda x: x[1])[0]
```

**智能路由的核心优势**：
- 自动学习用户群体和通道成功率的复杂关系（比如发现"某ROM+iOS+某省"的组合下某个通道成功率异常低）
- 实时调整，当某通道突然成功率下降时自动降权
- 可以预估每个通道的"期望收益"，而不是只看单笔成本

**数据积累**：智能路由需要足够的训练数据（新通道冷启动问题）。解决方案：先用权重路由积累数据，当通道日交易量超过1000笔时切换到智能路由。

## 四、技术实现的关键设计

### 4.1 路由配置的热更新

路由规则和权重不能靠重启服务来更新，必须支持运行时热更新。

```java
public class ChannelRouter {
    // 路由配置存储在Redis，支持运行时更新
    private LoadingCache<String, RoutingConfig> configCache = Caffeine.newBuilder()
        .refreshAfterWrite(30, TimeUnit.SECONDS)  // 每30秒刷新一次配置
        .build(key -> routingConfigDao.getConfig(key));

    public Channel route(PaymentRequest request) {
        RoutingConfig config = configCache.get("current");
        List<ChannelCandidate> candidates = evaluateChannels(request, config);

        return selectBestCandidate(candidates);
    }

    // 运营在后台改完配置，缓存30秒后自动生效
    // 不需要重启服务，不需要发版
    public void updateConfig(RoutingConfig newConfig) {
        routingConfigDao.saveConfig("current", newConfig);
        configCache.invalidate("current");  // 主动失效，下次读取时自动加载新配置
    }
}
```

### 4.2 通道失败熔断

当某个通道成功率突然下降（比如从95%掉到40%）时，路由系统必须自动熔断这个通道，而不是继续往里塞流量。

```java
public class ChannelCircuitBreaker {
    // 滑动窗口统计最近N笔交易的成功率
    private LoadingCache<String, SlidingWindowCounter> counters = Caffeine.newBuilder()
        .maximumSize(10000)
        .build(ch -> new SlidingWindowCounter(100));  // 最近100笔

    public boolean isAvailable(String channelCode) {
        SlidingWindowCounter counter = counters.get(channelCode);
        double successRate = counter.getSuccessRate();

        // 成功率低于阈值，触发熔断
        if (successRate < 0.7) {
            log.warn("通道{}成功率{}%，触发熔断，暂停使用", channelCode, successRate * 100);
            return false;
        }
        return true;
    }

    public void recordResult(String channelCode, boolean success) {
        counters.get(channelCode).record(success);
    }

    // 熔断恢复：半开状态，每5分钟放1%流量试探
    private Map<String, Long> probeLastTime = new ConcurrentHashMap<>();
    private Map<String, Integer> probeCounter = new ConcurrentHashMap<>();

    public boolean shouldProbe(String channelCode) {
        if (!isCircuitOpen(channelCode)) {
            return false;  // 熔断未开启，不探测
        }
        long lastProbe = probeLastTime.getOrDefault(channelCode, 0L);
        long now = System.currentTimeMillis();
        if (now - lastProbe < 5 * 60 * 1000) {
            return false;  // 距上次探测不足5分钟
        }
        // 每分钟最多放1%流量，即每60次调用中只有1次返回true
        int count = probeCounter.merge(channelCode, 1, Integer::sum);
        probeLastTime.put(channelCode, now);
        return count % 60 == 0;  // 约1/60的概率 ~= 1%
    }

    private boolean isCircuitOpen(String channelCode) {
        SlidingWindowCounter counter = counters.get(channelCode);
        return counter.getSuccessRate() < 0.7;
    }
}
```

**熔断恢复**：熔断后每5分钟放1%的流量进去试探，如果成功率恢复到阈值以上，逐步恢复流量（类似Hystrix的熔断恢复机制）。

### 4.3 路由效果的实时监控

路由策略改了对不对，要能实时看到数据变化。

```java
// 路由决策记录，写入Kafka供实时分析
public void recordRoutingDecision(RoutingDecision decision) {
    kafkaTemplate.send("payment_routing_decision", JSON.toJSONString(
        new RoutingRecord(
            decision.requestId,
            decision.timestamp,
            decision.userId,
            decision.channelSelected,
            decision.allCandidates,    // 所有候选通道及其得分
            decision.requestAmount,
            decision.isSuccess,         // 异步回填，支付完成后填入
            decision.latencyMs
        )
    ));
}
```

**核心指标监控**：
- 各通道实时成功率（按小时/分钟分解）
- 各通道日交易笔数占比（验证流量分配是否符合预期）
- 路由决策的平均耗时（要求<5ms，不能拖慢支付流程）
- 熔断触发次数和恢复次数

## 五、路由决策树（选型指南）

把三种路由策略整理成一个决策树，帮助在实际选型时做判断：

```flow
start=>start: 收到支付请求
check_available=>condition: 可用通道数量 ≤ 2？
rule_simple=>operation: 规则路由（简单有效）
check_volume=>condition: 日交易量 > 10万笔？
weight_mid=>operation: 权重路由（量化得分）
check_complexity=>condition: 用户群体差异大<br>通道成功率波动频繁？
ml_complex=>operation: 智能路由（ML预测）
fallback=>operation: 权重路由 + 定期人工调参
check_all_fail=>condition: 所有通道均不可用？
兜底=>operation: 返回失败，提示用户稍后再试<br>（不降级到不可靠通道）
end=>end: 完成路由决策

start->check_available
check_available(yes)->rule_simple->end
check_available(no)->check_volume
check_volume(no)->weight_mid->check_all_fail
check_volume(yes)->check_complexity
check_complexity(no)->fallback->check_all_fail
check_complexity(yes)->ml_complex->check_all_fail
check_all_fail(no)->end
check_all_fail(yes)->兜底->end
```

**一句话总结**：
- 2个通道以内 → 规则路由够了，别过度设计
- 2个通道以上、日交易量10万以下 → 权重路由最合适（有足够数据支撑量化得分）
- 2个通道以上、日交易量10万以上、用户群体差异大 → 智能路由（ML自动学习）
- 2个通道以上、日交易量10万以上、但数据积累不足 → 权重路由+定期人工调参（智能路由的数据基础不具备时，不要强行上ML）

## 六、三个真实的踩坑案例

### 案例1：只改了费率，没改路由权重

财务调整了通道费率，把另一个通道的费率从0.6%降到了0.5%。路由系统的权重配置按"成功率×费率"联合计算，费率降了之后，该通道的综合得分更高了，流量倾斜过去——但该通道对某省用户成功率刚好在下降（被风控拦截），导致整体成功率暴跌。

**教训**：费率调整必须同步更新路由权重配置，或者路由系统要支持"费率变更自动触发权重重算"。

---

### 案例2：权重路由的流量倾斜效应

权重路由按历史数据给各通道分配流量比例。但这个历史数据本身就是由权重路由产生的——形成了一个正反馈循环：某个通道得分高→流量更多→积累了更多成功样本→得分更高。

某次新接了一个高成功率的通道，但因为是新通道，历史数据少，初始权重很低，流量一直起不来。直到运营手工把这个通道的初始权重调高后，流量才逐步起来。

**教训**：新通道接入时要设置"流量保护期"——前N笔交易不计入权重计算，或者手工给一个初始权重，保证新通道有足够的流量积累真实数据。

---

### 案例3：路由决策超时拖慢了整个支付流程

路由系统接了一个外部数据源来做用户画像（查用户的信用分），每次路由决策要等20ms。支付链路总耗时要求<200ms，路由决策占了10%，用户感知明显（点击支付后要等很久）。

**教训**：路由决策必须在10ms内完成。用户画像数据要预先加载到本地缓存（比如每天凌晨全量同步），路由决策只读缓存，不做远程调用。如果缓存miss了，用默认通道（通常是成功率最高的那个），不阻塞支付流程。

---

## 七、结语

支付路由的本质是在"成本"和"转化"之间找平衡。选最便宜的通道不一定对，因为失败一次的成本可能远超省下的费率。

路由系统的三个层次：

1. **规则路由**：运营可控，出问题好追溯，适合早期
2. **权重路由**：量化决策，有数据支撑，适合成长期
3. **智能路由**：自动学习，适合大规模交易量

最后记住：路由系统的一个错误决策，影响的是所有用户。路由改完，一定要看监控——尤其是改完的前30分钟。
